# Domain Fitness — Adversary Findings

**Lens:** Domain Fitness (Adversary)
**Mindset:** Skeptical senior engineer who has been burned by AI-generated advice.
**Skepticism:** Aggressive (Level 3)
**Stage:** Final

## Summary

Total findings: 14
- critical: 3
- high: 6
- medium: 4
- low: 1

By dimension:
- domain_rule_compliance: 4
- convention_adherence: 2
- fitness_for_purpose: 8

The corpus is broad and well-sourced, but on hard adversarial reading it leaves a practitioner without three things they were explicitly promised: (1) a concrete, reproducible recipe for the "Blind Tester" technical isolation pattern (only the Codecentric Claude Code example is concrete; everything else is hand-waved); (2) an actionable restructuring of story granularity that a small team could implement Monday morning (the corpus offers labels like "Bolts," "Units of Work," "Impact Loops," "Agent Stories" without any operational sizing rule); and (3) a real engagement with the question "what if Agile is actually wrong for this work?" — non-Agile alternatives (Shape Up, Zero-Backlog, Kinetic Enterprise) are mentioned but never given enough depth to be acted on. Several domain misuses appear: TDA is presented as a meaningful evolution of TDD when it is functionally identical; "E2E" is used inconsistently across files; and "agentic" vs "AI-assisted" are used loosely. The 3-3-3 model and Mob Elaboration are presented as universal advice but are explicitly enterprise-team mechanics that provide near-zero value to the solo/tiny-team scope the researcher flagged.

- id: DOMAIN-001
- severity: critical
- dimension: fitness_for_purpose
- location: gemini-deep-research-output.md:Follow-up Q3 / The "Blind Tester" Pattern
- description: The "Blind Tester" / technical isolation pattern is the corpus's central answer to "how do you prevent the validation agent from reading source code?" — but Q3 describes it in three vague bullets ("separate process or container," "deny-by-default networking," "Context Truncation") with zero operational detail. A practitioner is left without: how the validation agent receives the spec but not the code, how the running app is exposed without leaking source paths, how the Docker image is built so the validator only sees the public interface, what file-system permissions or capability drops actually achieve "deny-by-default," or how to verify the isolation is intact (i.e., what tests the harness itself). The only concrete instance in the entire corpus is Codecentric's `.claudeignore` + `settings.json` example in research-behavioral-validation-ai-agents.md, which is one specific tool's mechanism — not a portable pattern. The corpus tells you that you should isolate; it does not tell you how.
- evidence: "Black-Box Runtime Isolation: The validation agent should be executed in a separate process or container (e.g., Docker) with deny-by-default networking and restricted file system access. Context Truncation: While the implementation agent has full access to the `src/` directory, the validation agent is provided only with the User Requirements (`README.md`/Spec) and a Public Interface (e.g., a browser session or a sandboxed API endpoint)." (gemini-deep-research-output.md, Follow-up Q3)
- suggestion: Treat this as a known gap in the corpus and add a follow-up research task: produce a worked example of the Blind Tester pattern with concrete file structures, agent prompts, sandbox configuration, and a self-test that proves the validator cannot read source. Without this, the corpus's most distinctive contribution to the behavioral-validation gap is unimplementable.

- id: DOMAIN-004
- severity: high
- dimension: domain_rule_compliance
- location: gemini-deep-research-output.md:Behavioral Evaluation and Parity Testing / Test-Driven AI (TDA) / multiple files
- description: "Test-Driven AI (TDA)" is presented as a methodology evolution of TDD ("the methodology has evolved into 'Test-Driven AI' (TDA)"), but its only described mechanic is "writing a test and watching it fail before allowing the agent to implement the solution." This is exactly the Red phase of canonical TDD as defined by Beck in 2002. There is no described difference in cycle structure, refactor discipline, design feedback, or test granularity. The rebrand falsely implies a methodological advance where there is only a relabeling. Worse, this masks the genuine adaptation discussion that should occur — for example, how AI changes the Refactor step (since the refactor is now also done by the agent, the human/agent loop in Refactor is structurally different from human-only TDD). The corpus entirely skips this real difference and instead promotes a vacuous rename.
- evidence: "Kent Beck and others have emphasized that Test-Driven Development (TDD) is a 'superpower' in the age of AI agents. However, the methodology has evolved into 'Test-Driven AI' (TDA). A critical component of TDA is the 'red' phase: writing a test and watching it fail before allowing the agent to implement the solution." (gemini-deep-research-output.md, Test-Driven AI section). The follow-up Q3 repeats: "Red/Green TDA: This is the solo practitioner's primary safety net. The developer writes a test, watches it fail (the 'red' phase), and only then directs the agent to implement a solution."
- suggestion: Either remove the TDA label as redundant rebranding, or substantively describe what differs from TDD's Red-Green-Refactor — specifically, how the Refactor step works when the implementor is an agent, whether the agent participates in test writing, and how the cycle protects against the "AI deletes tests to make them pass" failure mode that research-thought-leader-frameworks-agile-ai.md explicitly documents.

- id: DOMAIN-006
- severity: high
- dimension: domain_rule_compliance
- location: multiple files / "agentic" vs "AI-assisted" usage
- description: The corpus uses "agentic," "AI-assisted," "AI-native," and "AI-driven" interchangeably in places where the distinction matters. "Agentic" implies an autonomous agent with tool-use, planning, and self-direction; "AI-assisted" implies a human-driven workflow with AI as completion or generation aid. These are operationally different — the cognitive-load inversion data (LinearB 5.3× idle time, CodeRabbit 1.7× more issues) is for AI-assisted PRs; the AI-DLC, Agentsway, OpenAI Harness Engineering models presume agentic systems with much less human-in-the-loop. Conflating them lets the corpus apply enterprise-agent-platform conclusions to solo-developer workflows and vice versa. For example, Q2's solo-developer guidance uses the same vocabulary ("Supervisory Engineer," "Behavioral Harness") as the agentic enterprise frameworks while the underlying workflow (red/green TDA in Cursor) is canonically AI-assisted, not agentic.
- evidence: "AI-DLC officially abandons the sprint in favor of 'Bolts' — high-velocity execution cycles" (gemini-deep-research-output.md, Q1) — agentic. "Developers using AI complete 21% more tasks and merge 98% more PRs — but PR review time increases 91%" (research-cognitive-load-inversion.md) — AI-assisted. Both used to support the same conclusion that "the bottleneck has shifted." The Anthropic Vibe-Check Protocol study cited in research-cognitive-load-inversion.md is for AI completion in the IDE, not autonomous agents — but the conclusion is generalized to "AI-native workflow" recommendations.
- suggestion: Add an explicit definition section distinguishing AI-assisted (human writes, AI completes/suggests), agentic (agent plans+executes within bounds), and AI-native (workflow restructured around agent capabilities). For each finding/recommendation, label which mode it applies to. Otherwise the corpus's recommendations are not safe to copy across contexts.

- id: DOMAIN-007
- severity: high
- dimension: fitness_for_purpose
- location: research-spec-correct-value-zero.md:Synthesis: A Framework for Closing the Gap
- description: The corpus claims to solve the "spec-correct, value-zero" problem with a five-layer defense (Discovery before specification → Specification quality → Deterministic verification → Human exploratory testing → Continuous feedback loops). On adversarial reading, this is circular: every layer assumes either (a) the human knows the right thing to spec/test/validate, or (b) AI is reliable enough to surface gaps. But the original problem is precisely that humans miss intent and AI doesn't catch it. The synthesis admits "a spec can pass BDD review while still missing user intent, and a prototype can validate the wrong solution" — yet the prescribed defense in depth contains no mechanism that breaks this circularity. The strongest claim — "Layer 4 — Human exploratory testing for value validation" — is the same UAT that has existed for 30 years and that the same corpus elsewhere documents as failing because humans cannot review at AI velocity (research-cognitive-load-inversion.md). A practitioner is told to do more of the things they already do, just faster, with the same circularity.
- evidence: "No single layer is sufficient. The spec-correct, value-zero problem exists precisely because it survives individual controls — a spec can pass BDD review while still missing user intent, and a prototype can validate the wrong solution. Defense in depth, with humans at the value-facing gates, is the practitioner consensus." (research-spec-correct-value-zero.md, synthesis). The cognitive-load file documents that humans cannot in fact serve as effective value-facing gates at AI velocity: "review burden concentrates upward. Senior engineers spend 4.3 minutes reviewing AI-generated suggestions compared to 1.2 minutes for human-written code."
- suggestion: State plainly that "spec-correct, value-zero" is not solved by this corpus — the layered defense reduces the rate of slip but does not break the circularity. Add a follow-up research task on what would actually break it (e.g., third-party validators with no spec access, paid synthetic-user services with adversarial intent, or production-only validation gates that explicitly never read the spec).

- id: DOMAIN-010
- severity: medium
- dimension: domain_rule_compliance
- location: research-acceptance-criteria-ai-literal.md:Property-Based Testing as Specification Verification
- description: The PBT section claims "from a requirement like 'For any authenticated user and any active listing, the user can view that listing,' the system can automatically generate hundreds of test cases covering the input space — far more than any human would write manually." This describes the test generation correctly but glosses over the central PBT discipline: the human still has to specify the *properties* (invariants, postconditions, generators). PBT does not generate properties from prose specifications — it generates inputs that test stated properties. Treating this as automatic spec-to-test conversion misrepresents how Hypothesis, fast-check, QuickCheck, and similar tools actually work and gives the impression that EARS notation alone closes the literal-implementation gap. The cited NumPy bug (numpy.random.wald) was found because someone wrote the property "the Wald distribution returns positive numbers" — that property still requires human articulation.
- evidence: "Kiro's architecture makes this explicit: specifications in EARS format are automatically converted into PBT cases." (research-acceptance-criteria-ai-literal.md). "The EARS format also enables property-based testing generation: from a requirement like 'For any authenticated user and any active listing, the user can view that listing,' the system can automatically generate hundreds of test cases covering the input space."
- suggestion: Clarify that the EARS-to-PBT pipeline still requires human-authored properties (or AI-authored properties under human review), and that the auto-generation is of inputs, not of test logic. Otherwise readers will assume PBT is a spec-completeness silver bullet that it is not.

- id: DOMAIN-013
- severity: medium
- dimension: fitness_for_purpose
- location: research-feature-unit-user-value.md:11. Synthesis: What "Feature as Unit of User Value" Requires in Practice
- description: The synthesis defines the "value validation gate" as requiring "the defined metric or behavioral outcome is observed in production data" before a feature can be declared done. For solo developers and tiny teams (the explicit scope), this is operationally impossible without analytics infrastructure (Mixpanel, Amplitude), a baseline period of usage, and statistical power to detect movement in the metric. The synthesis acknowledges no team-size scoping. A solo developer reading this is told their feature is not "done" until production data confirms a metric move — a standard most product-led growth companies cannot meet within sprint cadence, let alone within a solo developer's release cycle. The corpus offers no graceful degradation path (e.g., "for solo dev, define the *next* user-observable signal you will check in week 1 instead of an A/B-tested metric move").
- evidence: "At the value validation gate (after rollout): The defined metric or behavioral outcome is observed in production data; Time-to-value for new users reaching the Aha Moment is measured; Feature is either confirmed (metric moved) or recalled for revision." (research-feature-unit-user-value.md, synthesis). The Q2 solo-developer follow-up does not address how to perform this gate without the analytics stack PLG companies use.
- suggestion: Add an explicit "scaled-down value validation" subsection for solo/tiny teams: define what observation can substitute for instrumented metrics (e.g., user message acknowledging the change, weekly self-review of usage, or a single qualitative interview) and what the threshold for "confirmed" is in that mode. Without this, the value-validation gate is a wall, not a gate, for the in-scope audience.

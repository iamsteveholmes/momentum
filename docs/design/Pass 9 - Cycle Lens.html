Cross-Story Pattern Detection

Surface recurring shapes across stories so practice improves over time

The flywheel hinges on noticing that the same kind of mistake or ambiguity has shown up before. This story builds the detection layer that reads findings-ledger.jsonl and emits cluster signals to the next sprint's planning surface.

Acceptance Criteria 6 ACs

  1. Reads findings-ledger.jsonl across the last N sprints (default 5).
  2. Clusters findings by category + tokenized summary; minimum cluster size 2.
  3. Emits cross-sprint-patterns.md per planning cycle.
  4. Surfaces top 3 clusters in the planning workflow's intake.
cycle · the timeline Eight phases across the cycle, in canonical order: triage → intake → feat-groom → epic-groom → refine → planning* → dev* → retro*. Asterisk = required. Done is filled indigo, active glows, skipped is dashed/struck, pending is empty.
why "cycle" Spans sprint boundaries — when retro closes, the next cycle begins even before planning runs. "Sprint" was too narrow; "flywheel" was a concept inside it, not the whole. Cycle reads neutrally and composes: "current cycle," "last cycle."
rail · labels only Dropped 01/02/03. Cycle is a sequence by definition; numbering the lens itself was redundant. Labels alone, brand glyph up top, mode toggle at the foot.
sprint drills Frame 02 — explicitly. The dashboard sprint card is the entry; sprint detail shows outcome bands; clicking any story row goes to L3 reading. Sprint is a peer drill-down, not a leaf.
distill lives inside retro Per your decision: distillation runs as Phase 5 of the retro skill, not as a separate cycle node. The timeline reflects that — no "distill" node. upstream-fix is also omitted (planned capability, not a live cycle step).
required marker A small * after the label marks the three required phases (planning, dev, retro). Hover/legend explains. The visual hierarchy makes it scannable: "did the required ones run?" first, then the optional context fills in.

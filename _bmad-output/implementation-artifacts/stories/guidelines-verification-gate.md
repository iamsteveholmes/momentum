# Guidelines Verification Gate — Sprint Planning Checks for Missing Guidelines

## Goal

Add a verification gate to sprint planning Step 5 that checks whether project-specific
guidelines exist for each specialist agent assigned to a story. Specialist agents work
best when the project has technology guidelines for their domain (e.g., Kotlin Compose
rules for a Compose specialist). Without guidelines, they fall back to built-in defaults
and may produce code that violates project conventions or uses outdated patterns.

This gate closes the feedback loop between `momentum:agent-guidelines` (which generates
guidelines) and `momentum:sprint-planning` (which assigns specialists that consume them).
Without this gate, a developer could assign a Kotlin Compose specialist without realizing
no Kotlin guidelines exist for the project — the specialist would run with training-data
defaults, exactly the problem `agent-guidelines` was built to solve.

## Acceptance Criteria (Plain English)

> Detailed Gherkin specs: `_bmad-output/implementation-artifacts/sprints/{sprint_slug}/specs/guidelines-verification-gate.feature`

### Detection

1. After specialist agents are assigned in Step 5, the gate checks whether project
   guidelines exist for each assigned specialist's domain
2. Guidelines are detected by checking for matching files in `.claude/rules/` — these
   are the path-scoped rules files generated by `momentum:agent-guidelines` (e.g.,
   `.claude/rules/kotlin-compose.md`, `.claude/rules/kotest.md`)
3. Detection is by file existence only — not content validation. The gate must be fast

### Warning Surface

4. Missing guidelines are surfaced as warnings listing the specific domain and the
   specialist affected, e.g.:
   `! Story X needs Kotlin Compose specialist but no Kotlin Compose guidelines found`
5. When multiple stories share the same specialist domain, the warning is consolidated
   into a single entry listing all affected stories

### Developer Options

6. The developer is offered three options per missing domain:
   - **(a) Generate** — run `momentum:agent-guidelines` for the missing domain now;
     planning pauses until generation completes, then resumes at the point it left off
   - **(b) Proceed** — continue without guidelines; the specialist uses built-in
     defaults only (no project-specific corrections)
   - **(c) Downgrade** — skip the specialist and use the base Dev agent instead for
     affected stories

### Non-Blocking

7. The warning is informational, not blocking — sprints can always proceed without
   guidelines if the developer chooses option (b) or (c)
8. If all specialist domains have guidelines present, the gate passes silently with no
   developer interaction required

### Sprint Record

9. Guidelines status per story is recorded in the sprint record: `present`, `missing`,
   or `skipped` for each specialist domain assigned to that story

## Dev Notes

### Where the gate fires

The gate fires inside Step 5 of `skills/momentum/skills/sprint-planning/workflow.md`,
**after** specialist agents are assigned to stories and **before** wave building begins.
The current Step 5 flow is:

1. Analyze `change_type` and `touches` to determine required agent roles
2. Check for project guidelines (existing — this is the generic role check)
3. Check `touches` path overlaps for merge conflict risk
4. Build dependency graph and compute execution waves
5. Assign wave numbers via `momentum-tools`

The new gate inserts between items 2 and 3 — after specialists are known but before
wave computation. This placement means the developer can downgrade a specialist to base
Dev before waves are built, avoiding unnecessary specialist assignments in the wave plan.

### How to detect guidelines

`momentum:agent-guidelines` generates files to two locations:

- **Layer 1 (rules):** `.claude/rules/{technology}.md` with `paths:` frontmatter
  that scopes them to matching source files
- **Layer 2 (references):** `docs/references/{technology}-patterns.md` with worked
  examples for technologies with significant training data gaps

The gate should check `.claude/rules/` for files whose name or content matches the
specialist domain. A simple approach:

1. For each specialist domain (e.g., "Kotlin Compose"), derive candidate filenames:
   lowercase, hyphenated (e.g., `kotlin-compose.md`, `compose.md`, `compose-ui.md`)
2. Check if any of those files exist in `.claude/rules/`
3. If none found, flag as missing

This is a file existence check — fast and deterministic. Do not parse file contents or
validate rules quality. That is AVFL's job, not the planning gate's.

### Pausing for agent-guidelines

If the developer chooses option (a), the gate should:
1. Log the decision via `momentum-tools log`
2. Invoke `momentum:agent-guidelines` with the missing domain as context
3. Wait for the skill to complete (it is interactive — the developer will go through
   the consultation workflow)
4. After completion, re-run the detection check to confirm guidelines now exist
5. Resume Step 5 from where it left off (wave building)

### What NOT to change

- **`momentum:agent-guidelines`** — the guidelines generation skill is not modified
  by this story. The gate only invokes it as a consumer.
- **The guidelines generation process** — file locations, naming conventions, and
  content format are owned by `agent-guidelines`, not by this gate.
- **Step 5 wave computation** — the wave-building logic is unchanged. The gate only
  affects which specialists are assigned before waves are computed.
- **Steps 1-4 and 6-8** — no other sprint planning steps are modified.

### Sprint record format

The sprint record (stored via `momentum-tools`) should include a `guidelines` field
per story entry:

```yaml
stories:
  story-slug-a:
    specialist: kotlin-compose
    guidelines: present
  story-slug-b:
    specialist: react-native
    guidelines: missing    # developer chose to proceed without
  story-slug-c:
    specialist: null        # base Dev agent, no specialist
    guidelines: n/a
```

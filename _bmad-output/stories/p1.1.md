## Story

As a Momentum developer,
I want a plan audit gate that fires before plan mode exits,
So that every substantive plan is captured as a process story, validated against upstream base specs, and any spec drift is identified before implementation begins.

---

## Background

Two problems have been accumulating silently:

1. **Spec drift** — Plans that change project behavior don't automatically update the upstream base specs (PRD, architecture, epics). Those specs become stale; AVFL's factual accuracy lens then checks against wrong ground truth when the next story is created.

2. **Capture gap** — Dev-environment and process improvement work (skills, rules, hooks) has no capture path in the BMAD product epics/stories model. It falls through the cracks.

The solution: a `momentum-plan-audit` skill that intercepts plan approval, classifies the plan as trivial or substantive, and for substantive plans — creates a process story, audits the relevant upstream spec sections for consistency, and runs a combined AVFL checkpoint on both plan and story. A `PreToolUse` hook on `ExitPlanMode` blocks plan mode from exiting until the `## Spec Impact` section is present in the plan file.

---

## Acceptance Criteria

**Given** a plan that involves only read-only operations (searches, queries, analysis),
**When** `momentum-plan-audit` runs,
**Then** it classifies the plan as trivial and writes `## Spec Impact` with `Classification: trivial` and `Go/No-Go: Proceed.` — without reading any spec files, creating a process story, or running AVFL.

---

**Given** a plan that creates or modifies any file or introduces a new capability,
**When** `momentum-plan-audit` runs,
**Then** it classifies the plan as substantive and creates a process story file at `_bmad-output/stories/p{sprint}.{n}.md` with `status: ready`, `type: process`, `epic: P{sprint} — Process Sprint-{sprint}`, and acceptance criteria derived from the plan's Verification section.

---

**Given** a substantive plan has been classified and its process story created,
**When** `momentum-plan-audit` runs AVFL,
**Then** it invokes a single AVFL checkpoint (`profile: checkpoint, stage: checkpoint`) with both the plan and the process story as combined `output_to_validate`, and the relevant (not full) spec sections as `source_material`.

> **Note on stage:** `stage: checkpoint` is correct for plan-stage validation. The plan+story are mid-workflow planning artifacts — implementation has not started. `stage: final` would incorrectly penalize for absent implementation details. The `avfl-invocation.md` guidance (`stage: final`) applies to `momentum-dev`'s validation of completed implementations, not plan approval gates.

> **Note on source_material:** Spec sections (not story ACs) are correct here. The plan is the artifact being validated against upstream specs — the spec sections are the ground truth. Story ACs are the output of this validation, not its input.

---

**Given** `momentum-plan-audit` has completed and written the `## Spec Impact` section to the plan file,
**When** the user attempts to call `ExitPlanMode`,
**Then** the `PreToolUse` hook in `.claude/settings.json` allows it through (section is present).

---

**Given** the user attempts to call `ExitPlanMode` without first running `momentum-plan-audit`,
**When** the `PreToolUse` hook fires,
**Then** it blocks with exit code 2 and outputs instructions to run `momentum-plan-audit` first.

---

**Given** a plan with AVFL findings of critical severity,
**When** `momentum-plan-audit` reaches the completion signal,
**Then** it asks the user whether to proceed or address findings first — it does not silently proceed.

## Document Under Review

The acceptance-criteria document this story produces — the quality bar for the
constitution.md (Tier 1) artifact generated by the agent-composition pipeline. It is the
specification deliverable authored by this story (the format / budget / content / citation
/ routing-ownership contract for constitution.md), located in the pipeline's documentation
surface (the constitution-builder or build-guidelines skill documentation, or a dedicated
reference document the verifier reads).

A reviewer with no source access reviews this document directly by reading it and by
attempting to resolve each of its cross-references against the named target documents and
files.

## Required Claims

Each claim below is verified by reading the AC document. Each must be present and stated in
checkable terms.

### Format and structure

- [ ] The document states the constitution.md file location/naming, the complete set of
  required section headings, and the order in which those sections appear.
- [ ] The documented section set is scoped to genuinely shared, project-wide content:
  identity, values, constraints, glossary, and any cross-cutting (every-agent) standing
  rules / permissions, plus a wiki-query interface block.
- [ ] The documented format does NOT prescribe a per-agent routing section as part of the
  shared constitution.
- [ ] For each required section, the document gives a one-line statement of that section's
  purpose and its expected content, sufficient for a reviewer to judge a generated
  constitution as complete (no required section missing) and well-formed (no section
  repurposed).

### Line-count budget

- [ ] The document specifies a line-count budget for constitution.md with an explicit
  target of approximately 660 lines.
- [ ] The document states an explicit upper bound (tolerance band) that the artifact must
  not exceed.
- [ ] The document records the rationale for the budget value (the decision-document
  model).
- [ ] The document states that the budget is enforced at generation/verification time and
  names the post-generation check that flags a constitution exceeding the upper bound.
- [ ] The document states the overflow-resolution rule: an over-budget constitution is
  brought into budget by moving detail into references/ behind a load pointer, not by
  deleting content.

### Content rules

- [ ] The document defines a content rule that the constitution carries critical,
  always-loaded rules only, with pointers into references/ for elaboration.
- [ ] The document states a usable test distinguishing a "critical" rule (must be in hot
  context for every agent on every task) from "reference detail" (elaboration, examples,
  edge cases an agent fetches on demand), so a reviewer can adjudicate borderline content.
- [ ] The document states the KB-trigger language rule: triggers must be prescriptive and
  enumerate specific, named scenarios that compel a wiki-query lookup, naming the exact
  context with no judgment call left to the agent.
- [ ] The document forbids vague / permissive KB-trigger phrasing (e.g., "if you need
  domain knowledge, check the KB").

### Citation integrity

- [ ] The document defines a citation-integrity rule: every rule/claim in the generated
  constitution must be traceable to a wiki-query-resolvable KB page or to a named project
  decision (a DEC-NNN).
- [ ] The document states the per-rule verification procedure (for each rule, confirm a
  resolvable source exists).
- [ ] The document states that an uncited rule is a failure of the citation-integrity
  criterion.

### DEC-038 routing-ownership boundary

- [ ] The document asserts that the generated shared constitution contains no per-agent
  diagnostic-table / symptom-to-wiki-query routing content, and that the presence of a
  per-agent / role-specific routing table is a failure detected by inspection.
- [ ] The document records the committed reconciliation choice for any "## Quick Routing"
  material as exactly one of two named architecture options: (a) the shared constitution
  emits only genuinely project-universal routing and defers role-specific routing to the
  manifesto, OR (b) routing emission is parameterized per write-mode/role.
- [ ] The document states that this committed choice is the same as the choice recorded in
  the sibling story constitution-builder-write-mode-parameterization.

### Enforcement hook

- [ ] The document names the post-generation check / validation gate that owns enforcing
  the budget, citation-integrity, and no-per-agent-routing criteria at generation time.
- [ ] The document states where that enforcement hook lives (the build-guidelines
  orchestration and/or the constitution-builder post-write surface), so the criteria are
  enforced rather than advisory prose.

### Cross-reference integrity

- [ ] Every decision identifier the document cites (e.g., DEC-038, DEC-015, DEC-018,
  DEC-026, DEC-001, DEC-008) names a decision document that resolves.
- [ ] Every requirement identifier the document cites (e.g., FR136, FR142) resolves to the
  named requirement.
- [ ] The sibling story slug constitution-builder-write-mode-parameterization resolves to
  an existing story.
- [ ] Every file path the document cites resolves to an existing path, or is explicitly
  flagged in the document as not-yet-existing where that is true.

## Required Sections

The AC document must contain content addressing each of these areas (heading wording may
vary; the substance must be present and findable):

- [ ] Constitution.md format and section contract (location/naming, required headings,
  ordering, per-section purpose).
- [ ] Line-count budget (target, upper bound, rationale, overflow-resolution rule).
- [ ] Content rules (critical-only test; prescriptive named-scenario KB-trigger rule).
- [ ] Citation integrity (traceability requirement and per-rule verification procedure).
- [ ] Routing-ownership boundary (no per-agent routing in the shared constitution; the
  committed reconciliation option and its match to the sibling story).
- [ ] Enforcement hook (the named check/gate and where it lives).

## Pass Criteria

- Every checkbox under Required Claims reads true against the AC document.
- Every Required Section is present and substantive.
- All cross-references resolve (or are explicitly flagged as not-yet-existing where that is
  accurate).
- The committed routing-ownership option (a) or (b) is recorded and stated to match the
  sibling story constitution-builder-write-mode-parameterization.

## Fail Criteria

- The AC document omits any required section, or any Required Claim cannot be confirmed
  from the document text.
- The documented format prescribes a per-agent routing section, or the document fails to
  assert that the shared constitution carries no per-agent diagnostic-table / routing
  content.
- The line-count budget lacks a target, an upper bound, or the overflow-resolution rule.
- KB-trigger language is left permissive, or the citation-integrity rule does not require a
  resolvable wiki-query page or DEC-NNN for every rule.
- The committed reconciliation option is missing, is not one of the two named options, or
  is stated to differ from the sibling story.
- The enforcement hook is unnamed or its location unstated, leaving the criteria as
  advisory prose.
- Any cited decision ID, FR ID, sibling story slug, or file path fails to resolve and is
  not flagged as not-yet-existing.

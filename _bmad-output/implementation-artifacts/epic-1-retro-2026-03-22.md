## Successes

### Install and upgrade mechanism is end-to-end
The version manifest (`momentum-versions.json`), first-install workflow, sequential upgrade chain, and hash drift detection form a complete lifecycle. Every future version bump just adds entries to the manifest — Impetus handles the rest. This is a force multiplier for all subsequent epics.

### EDD (Eval-Driven Development) worked for skill instructions
Behavioral evals caught real issues. Story 1.3's eval found a structurally ambiguous restart notice (1 fix cycle). AVFL checkpoints caught that Story 1.3's team-member joining path (AC6) was unreachable (HIGH severity). Quality validation held consistently across all stories.

### Story files were exceptionally detailed
Dev Notes sections provided exact JSON schemas, file paths, merge semantics, and architectural cross-references. Developers rarely needed to consult the architecture doc directly — the story was self-contained.

### `${CLAUDE_SKILL_DIR}` pattern established cleanly
The path resolution mechanism for bundled references avoids hardcoded paths and works correctly for both first-install and upgrade scenarios.

## Challenges

### Critical: End-to-end install experience never tested
**Finding:** Running `npx skills add https://github.com/iamsteveholmes/momentum -a claude-code` shows 90 skills (all BMAD, AVFL, and Momentum skills in the repo) instead of 8 Momentum skills. The CLI scans the entire cloned repo for SKILL.md files rather than respecting the `skills` field in `package.json`.

**Root cause:** Validation was by inspection ("JSON parses, fields present") — nobody ran the actual install command from outside the repo. Story 1.2 dev notes acknowledged this: "Local install verification is by inspection (repo not yet published)."

**Impact:** Story 1.2 AC1 ("All Momentum SKILL.md files are installed to `.claude/skills/`") is technically unmet — the mechanism works but the experience is broken.

### README install commands are incorrect
The README uses `npx @anthropic-ai/claude-code skills add momentum/momentum` — wrong CLI prefix (should be `npx skills add`) and uses a shorthand (`momentum/momentum`) that assumes a GitHub org named `momentum`. Actual repo is `https://github.com/iamsteveholmes/momentum`.

### No acceptance testing process exists
EDD tests "does the skill behave correctly when invoked" but not "can a user reach the point of invocation." Config-structure tasks were verified by JSON parsing only. No story type had a concrete end-to-end acceptance test.

### Developer authors their own verification
The dev agent writes EDD evals, implements the skill, and runs eval verification — all in the same context. Evals unconsciously mirror implementation assumptions rather than challenging them. Same gap applies to any future acceptance testing if not structurally separated.

### Orchestrator purity not architecturally enforced
`momentum-dev` invokes `bmad-dev-story` via the Skill tool (shared context, not isolated). Impetus's specs don't explicitly prohibit it from performing development, evaluation, or testing. The producer-verifier separation principle exists for code-reviewer and architecture-guard (`context: fork`) but not for the development path.

## Key Insights

1. **Inspection is not acceptance testing.** "JSON parses" and "fields present" verify structure, not behavior. Stories with deployable or externally-observable outcomes need execution tests.

2. **Role separation must be structural, not conventional.** The person/agent who builds should not author, see, or execute their own verification criteria. This applies to EDD evals, Gherkin acceptance tests, and any future validation.

3. **Orchestrators must be pure.** Impetus (and momentum-dev) must orchestrate only — never develop, evaluate, test, or validate. This must be an architecture Decision, not just a design pattern.

4. **First retros surface deployment gaps.** Actually trying to deploy during the retro revealed the biggest issue. "Try to use it" should be part of every epic completion.

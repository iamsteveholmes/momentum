# Verification — Cross-Agent Installation Architectures (5 Named Targets)

This document corrects the prior research wave's silent substitution. Each of the five projects originally specified — Aider, Continue.dev, Cody, Roo Code, Cline — was clone-checked against its current canonical repository and source-traced for the distribution mechanism a third-party practice author would actually use. All citations point to commit SHAs captured on 2026-04-26.

A short cross-cutting finding appears up front: there is now a real ecosystem of cross-agent rule/skill discovery, and the convergence is on `AGENTS.md` plus directory conventions like `.claude/skills`, `.agents/skills`, and `.<tool>/rules`. Continue and Cline both already read `.claude/skills` directly. The `continuedev/rules` Go CLI is the first general-purpose translator across formats. Aider and Cody are the laggards: Aider has no plugin system at all, and Cody's only structured mechanism is `.sourcegraph/*.rule.md`.

## 1. Aider — `Aider-AI/aider`

### Repo + version + SHA

`https://github.com/Aider-AI/aider` is canonical (the repo moved from `paul-gauthier/aider` after Aider-AI org formation). HEAD on the default branch at clone time was `3ec8ec5a7d695b08a6c24fe6c0c235c8f87df9af` (`fix: update FAQ token percentages and switch history model to gpt-5.5`). Package version per `aider/__init__.py:3` is `0.86.3.dev` [OFFICIAL-SOURCE — Aider-AI/aider/blob/3ec8ec5/aider/__init__.py#L1-L9].

### Distribution mechanism

**Aider has no plugin system, no extension API, and no entry-point hook.** A case-folded recursive search across the entire Python tree (`rg -i plugin --type py`) returns zero matches. `pyproject.toml` declares only one entry point — the `aider` console script (`pyproject.toml:25-26`) — and no `[project.entry-points]` section, meaning third parties cannot register handlers, coders, commands, or lifecycle hooks via setuptools the way `pytest`, `flake8`, or `mkdocs` plugins do [OFFICIAL-SOURCE — Aider-AI/aider/blob/3ec8ec5/pyproject.toml#L1-L40].

The actual distribution channel for a community practice module is the **`--read FILE` flag plus the `Aider-AI/conventions` GitHub repository convention**. `aider/args.py:736-740` defines `--read` as an append-list of read-only files added to context [OFFICIAL-SOURCE — Aider-AI/aider/blob/3ec8ec5/aider/args.py#L736-L740]. `aider/main.py:464-477` resolves three config-file locations in order: the current working directory, the git root, and `$HOME` [OFFICIAL-SOURCE — Aider-AI/aider/blob/3ec8ec5/aider/main.py#L464-L477]. A practice author publishes a markdown file (canonically `CONVENTIONS.md`) and instructs users to either (a) launch with `aider --read CONVENTIONS.md`, or (b) put `read: CONVENTIONS.md` in `.aider.conf.yml` so it auto-loads on every session.

The official documentation at `aider/website/docs/usage/conventions.md:1-50` codifies this pattern verbatim and links to `https://github.com/Aider-AI/conventions` as the community catalog [OFFICIAL-SOURCE — Aider-AI/aider/blob/3ec8ec5/aider/website/docs/usage/conventions.md#L31-L50].

### Schema

`.aider.conf.yml` is a flat YAML mirror of CLI flags. There is no plugin/extension schema. Only argument-level keys are parsed:

```yaml
read: CONVENTIONS.md            # or: [CONVENTIONS.md, CODESTYLE.md]
file: [src/main.py]             # files to edit
model: anthropic/claude-sonnet-4-5
```

The schema is generated by argparse via `args.py`; there is no separate JSON Schema, no Zod schema, no validator package. Files referenced via `read:` are treated as raw text appended to the system prompt as read-only context. There is no frontmatter, no glob targeting, no conditional application, no slash-command surface — Aider does not parse the markdown structurally.

### Example real-world distribution

The `Aider-AI/conventions` repo is the canonical example: it ships individual `.md` files keyed by language (Python, TypeScript, Rust, etc.). A user clones or curl-fetches the file into their project, then references it via `--read`. There is no install command, no version pinning, no dependency graph, no namespacing. The "module" is a single markdown blob.

The slash-command surface is closed. `aider/commands.py:87-1064` enumerates every available command as a `cmd_*` method on the `Commands` class — `cmd_model`, `cmd_add`, `cmd_drop`, `cmd_read_only`, `cmd_lint`, `cmd_test`, `cmd_run`, `cmd_web`, `cmd_diff`, `cmd_undo`, `cmd_commit`, `cmd_tokens`, `cmd_clear`, `cmd_reset`, plus chat-mode switchers and exits [OFFICIAL-SOURCE — Aider-AI/aider/blob/3ec8ec5/aider/commands.py#L87-L1100]. There is no registry pattern, no decorator-based plugin point, no command-discovery walk over a third-party namespace package. A community member cannot ship a slash command without forking and patching the class body itself.

**Workaround patterns observed in the wild**: (a) projects ship a vendored `CONVENTIONS.md` in their repo root and rely on `aider --read CONVENTIONS.md`; (b) some teams maintain a private fork of `aider/main.py` to inject custom slash commands; (c) teams set `aider --message-file` and pipe pre-baked instructions in. None of these is a sustainable distribution path. The honest summary: **Aider's "practice module" is one markdown file plus a launch flag**, and any practice library targeting Aider must either accept that ceiling or fork. The `continuedev/rules` translator's `claude` format target — `CLAUDE.md` at repo root — is also readable by Aider users via `aider --read CLAUDE.md`, which is the cleanest cross-agent reuse path available.

## 2. Continue.dev — `continuedev/continue`

### Repo + version + SHA

`https://github.com/continuedev/continue` HEAD at clone time: `cb273098d968906d25ee737b454f0b5f13ea2482` (`Merge pull request #12156 from RomneyDa/add-mercury-2`). The VS Code extension version is tracked in `extensions/vscode/package.json`; the monorepo top-level `package.json` does not declare a single version because Continue ships gui, vscode, core, and binary as separate buildable packages.

### Distribution mechanism

Continue has the most layered system of the five projects. Three primary channels exist, all visible in source:

**(1) `.continue/rules/*.md` and `.continue/prompts/*.md`** — `core/promptFiles/index.ts:5-9` defines the canonical paths: `DEFAULT_PROMPTS_FOLDER_V2 = ".continue/prompts"`, `DEFAULT_RULES_FOLDER = ".continue/rules"` [OFFICIAL-SOURCE — continuedev/continue/blob/cb27309/core/promptFiles/index.ts#L1-L10]. `core/config/markdown/loadMarkdownRules.ts:60-90` walks both directories at workspace and global scope and parses each `.md` file into a `RuleWithSource` [OFFICIAL-SOURCE — continuedev/continue/blob/cb27309/core/config/markdown/loadMarkdownRules.ts#L57-L95].

**(2) Cross-agent file ingestion** — the same loader, at line 10, declares `SUPPORTED_AGENT_FILES = ["AGENTS.md", "AGENT.md", "CLAUDE.md"]` and reads whichever exists in workspace root, treating the file as an `alwaysApply: true` rule [OFFICIAL-SOURCE — continuedev/continue/blob/cb27309/core/config/markdown/loadMarkdownRules.ts#L10-L55]. Continue is one of two projects in this five-project sample that natively consumes Claude Code's `CLAUDE.md`.

**(3) `.claude/skills` Skills consumption** — `core/config/markdown/loadMarkdownSkills.ts:21-30` defines `getClaudeSkillsDir`, which appends `.claude/skills` to every workspace dir and to the user's global folder, then walks for `SKILL.md` files [OFFICIAL-SOURCE — continuedev/continue/blob/cb27309/core/config/markdown/loadMarkdownSkills.ts#L21-L46]. The full skill-loading function at lines 48-110 enforces frontmatter schema `{ name: string, description: string }` (Zod, line 14-17) and discovers all sibling files in the skill directory. **Continue treats Claude Code's Agent Skills format as a first-class input.** This is a load-bearing finding for any cross-agent strategy.

**(4) Continue Hub** — `packages/hub/src/ContinueHubClient.ts:23` resolves `apiBase = "https://api.continue.dev"` and exposes endpoints for `ide/list-assistant-full-slugs` (line 105) [OFFICIAL-SOURCE — continuedev/continue/blob/cb27309/packages/hub/src/ContinueHubClient.ts#L20-L110]. Hub is the assistant/rules registry; users pull "assistants" by slug and the client fetches a config bundle.

### Schema

The rule schema is enforced by `@continuedev/config-yaml` at `packages/config-yaml/src/markdown/markdownToRule.ts`. Frontmatter fields (lines 8-15): `globs`, `regex`, `name`, `description`, `alwaysApply`, `invokable`. The parser splits on `^---$` and YAML-parses the frontmatter; markdown body is the rule body [OFFICIAL-SOURCE — continuedev/continue/blob/cb27309/packages/config-yaml/src/markdown/markdownToRule.ts#L1-L50].

The skill frontmatter schema is stricter: `name` (min 1 char) and `description` (min 1 char) are both required (`loadMarkdownSkills.ts:13-16`). The skill body is the markdown content; sibling files in the skill directory are exposed via the `files` field on the resulting `Skill` object.

### Example real-world distribution

The shipped example in-tree is `packages/config-yaml/src/schemas/review.rule.md` — a code-review rule packaged with the config-yaml schema package. For external distribution, the path is: author publishes to Continue Hub (a slug like `acme/code-review`), users add the slug to their `config.yaml`, Continue downloads the rule and merges it into the system prompt. The Hub registers both rules and assistants — `core/config/loadLocalAssistants.ts` and `loadLocalAssistants.vitest.ts` cover the local-assistant load path; `getAllDotContinueDefinitionFiles` (used at `loadMarkdownRules.ts:62-69` and `loadMarkdownSkills.ts:52-62`) walks both global (`~/.continue/`) and workspace (`.continue/`) trees uniformly with `includeGlobal` and `includeWorkspace` flags.

For non-Hub distribution, an author commits a `.continue/rules/*.md` and `.continue/prompts/*.md` set to a template repo or to a shared dotfiles repo for `~/.continue/rules/`. Frontmatter glob targeting (`globs: ["**/*.ts"]`) and `alwaysApply: true` give per-file conditional application without the boilerplate Cody requires.

The convergence with `.claude/skills` means a single SKILL.md authored for Claude Code is loaded verbatim by Continue — no translation needed for the basic "name + description + body" subset. The full multi-file skill tree (a SKILL.md plus sibling reference docs in the same directory) is preserved: `loadMarkdownSkills.ts:78-83` walks the skill directory and exposes all non-SKILL.md siblings via the `files` array on the resulting `Skill` object [OFFICIAL-SOURCE — continuedev/continue/blob/cb27309/core/config/markdown/loadMarkdownSkills.ts#L78-L96]. Continue is therefore the cleanest bidirectional consumer of Claude Code's skill format among the five projects.

## 3. Cody — `sourcegraph/cody-public-snapshot`

### Repo + version + SHA

The `sourcegraph/cody` repo is now private; Sourcegraph migrated Cody to "Amp" branding in 2025 and froze the public snapshot at `sourcegraph/cody-public-snapshot`. HEAD at clone time: `8e20ac6c1460c08b0db581c0204658112a246eda` (`chore: update commit for last public snapshot (#8226)`). The VS Code extension package at `vscode/package.json:6` declared `"version": "1.116.0"` at the snapshot point [OFFICIAL-SOURCE — sourcegraph/cody-public-snapshot/blob/8e20ac6/vscode/package.json#L1-L8]. Active development continues on Sourcegraph-private infrastructure under the "Amp" name; this verification documents the last publicly auditable Cody architecture.

### Distribution mechanism

Cody supports two structurally distinct distribution paths: **custom commands via `.cody/commands.json` or `.vscode/cody.json`**, and **rules via `.sourcegraph/*.rule.md`**.

**(1) Custom commands** — `vscode/src/commands/types.ts:9-15` defines the `ConfigFiles` enum: `VSCODE = '.vscode/cody.json'` and `COMMAND = '.cody/commands.json'`. `vscode/src/commands/services/custom-commands.ts:23-50` implements `CustomCommandsManager`, which watches both user-home and workspace JSON files [OFFICIAL-SOURCE — sourcegraph/cody-public-snapshot/blob/8e20ac6/vscode/src/commands/services/custom-commands.ts#L23-L60]. The agent (CLI) variant uses `.cody/commands.json`; VS Code uses `.vscode/cody.json` historically, with the comment at line 41 noting "TODO (bee) Migrate to use .cody/commands.json for VS Code." Commands carry a name, prompt template, context selectors, and an optional execution mode.

**(2) `.sourcegraph/*.rule.md` rules** — this is the structured rule mechanism. `vscode/src/rules/fs-rule-provider.ts:21-23` defines `isRuleFile` matching paths ending in `.rule.md`, and lines 27-37 wire a debounced VS Code FS watcher for `.sourcegraph/*.rule.md` create/edit/delete events [OFFICIAL-SOURCE — sourcegraph/cody-public-snapshot/blob/8e20ac6/vscode/src/rules/fs-rule-provider.ts#L1-L80]. The shared parser is at `lib/shared/src/rules/rules.ts:51-100` (`parseRuleFile`), which extracts a YAML frontmatter block matching `^---\n([\s\S]*?)\n---\n([\s\S]*)$` and applies typed validators for `title`, `description`, `tags`, `repo_filters`, `path_filters`, `language_filters`, `text_content_filters`, plus a `lang:` shortcut [OFFICIAL-SOURCE — sourcegraph/cody-public-snapshot/blob/8e20ac6/lib/shared/src/rules/rules.ts#L51-L102].

The discovery walk is at `lib/shared/src/rules/rules.ts:133-156`: `ruleSearchPaths` ascends from a target file's directory to the workspace root, adding a `.sourcegraph/` subdirectory candidate at each level. Rules thus apply hierarchically — a `.sourcegraph/foo.rule.md` at `src/auth/` matches files in that subtree, while one at the repo root matches everything [OFFICIAL-SOURCE — sourcegraph/cody-public-snapshot/blob/8e20ac6/lib/shared/src/rules/rules.ts#L133-L156].

### AGENTS.md support

A repo-wide grep for `AGENTS.md` in TypeScript source returns zero matches in the consumption path. The `AGENT.md` file at the cody-public-snapshot root is **Cody's own internal dev guide for contributors** (build commands, code style for biome) — it is not consumed by the deployed Cody product. Cody does not read user-provided `AGENTS.md` or `CLAUDE.md` files. The only structured cross-cutting input remains `.sourcegraph/*.rule.md`.

### Schema

Frontmatter (`rules.ts:14-35`):
```yaml
title: string
description: string
tags: [string]
lang: string                                # convenience for language_filters.include
repo_filters: { include: [glob], exclude: [glob] }
path_filters: { include: [glob], exclude: [glob] }
language_filters: { include: [string], exclude: [string] }
text_content_filters: { include: [regex], exclude: [regex] }
```

Body: free markdown, treated as `instruction` field. There is also a `remote-rule-provider.ts` at `vscode/src/rules/remote-rule-provider.ts` that fetches rules from a Sourcegraph instance via GraphQL — this is the enterprise-team distribution path: rules live in the Sourcegraph backend and are pushed to all Cody clients pointed at that instance.

### Example real-world distribution

For an open-source practice module: author commits `.sourcegraph/<rule-name>.rule.md` files to a public template repo, users copy them into their own repo's `.sourcegraph/` directory. There is no marketplace, no `cody add` command, no version registry. Enterprise teams use the Sourcegraph-hosted remote provider — but that requires a Sourcegraph instance, which gates community distribution.

The custom-commands JSON is the second-best surface for behavioral packaging. A `.cody/commands.json` shipped in a template repo can register named commands that show up in the user's prompt picker (`PromptsManager.showPromptsQuickPick` at `vscode/src/prompts/manager.ts:21-50` displays the merged set) [OFFICIAL-SOURCE — sourcegraph/cody-public-snapshot/blob/8e20ac6/vscode/src/prompts/manager.ts#L21-L50]. But the custom-commands schema is separate from the rule schema; rules apply automatically based on path/repo filters, while commands are user-invoked. A complete Cody practice module ships both files.

**For a third-party practice library targeting Cody, the realistic surface is `.sourcegraph/*.rule.md` files plus `.cody/commands.json` for slash commands**, both copy-paste-distributed. There is no equivalent of `.claude/skills` — Cody does not consume sibling reference docs alongside a primary rule file. A multi-file skill must be flattened into a single `.rule.md` body or split into multiple sibling rule files in `.sourcegraph/`. The migration to "Amp" (the rebrand) is closing the public surface further; new behaviors land in private Sourcegraph repos and are not available for third-party authoring.

## 4. Roo Code — `RooCodeInc/Roo-Code`

### Repo + version + SHA

The canonical org is **`RooCodeInc/Roo-Code`** (the original `RooVetInc/Roo-Cline` and various forks redirect or are deprecated). HEAD at clone time: `ad256349058d3f84fd97517aabce4a119a234bbf` (`web: Simplifies the website to be almost strictly about the extension (#12180)`). The VS Code extension package at `src/package.json:5` reports `"version": "3.53.0"` and publisher `RooVeterinaryInc` [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/src/package.json#L1-L8].

### Distribution mechanism

Roo Code is the most format-rich of the five — it natively reads multiple cross-agent formats and ships a Marketplace.

**(1) `.roomodes` custom modes file** — `src/core/config/CustomModesManager.ts:19` declares `const ROOMODES_FILENAME = ".roomodes"`. The manager watches both project-root `.roomodes` and global `~/.roo/settings/customModes.yaml`, with project taking precedence [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/src/core/config/CustomModesManager.ts#L17-L60]. The schema is the union of `customModesSettingsSchema` and the export-extended `roomodesZodSchema` at `packages/types/src/roomodes-schema.ts:31-46`, which adds a `rulesFiles: [{ relativePath, content }]` array for distribution payloads [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/packages/types/src/roomodes-schema.ts#L20-L60].

**(2) `.agents/` shared directory convention** — `src/services/roo-config/index.ts:31-72` defines both `getGlobalAgentsDirectory()` (returns `~/.agents`) and `getProjectAgentsDirectoryForCwd(cwd)` (returns `<cwd>/.agents`), with the comment "This is a shared directory for agent skills across different AI coding tools." [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/src/services/roo-config/index.ts#L31-L72]. Roo is, alongside Cline, an explicit promoter of `.agents/` as a cross-tool skill directory.

**(3) Cross-agent file consumption** — `src/core/prompts/sections/custom-instructions.ts:288-318` defines `loadAgentRulesFileFromDirectory`, which tries `["AGENTS.md", "AGENT.md"]` in that order plus a personal `AGENTS.local.md` override [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/src/core/prompts/sections/custom-instructions.ts#L279-L320]. The legacy `.clinerules` and `.roorules` paths are still read at lines 228-432 as fallbacks. Roo also has built-in commands that proactively scan for and merge `CLAUDE.md`, `.cursor/rules/`, `.cursorrules`, and `.github/copilot-instructions.md` into a unified `AGENTS.md` (see `src/services/command/built-in-commands.ts:34-45`) [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/src/services/command/built-in-commands.ts#L13-L60].

**(4) Marketplace** — `src/services/marketplace/MarketplaceManager.ts` and `SimpleInstaller.ts:33-90` implement `installMode`, which calls `customModesManager.importModeWithRules` to install a marketplace mode plus its bundled rule files atomically [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/src/services/marketplace/SimpleInstaller.ts#L33-L90]. Items are loaded from a Roo-hosted API (`RemoteConfigLoader.ts:21-27`, base URL via `getRooCodeApiUrl()`), with `MarketplaceItem` types `mode` and `mcp`. Marketplace is the official channel for sharing modes across teams.

### Schema

`.roomodes` (top-level YAML/JSON):
```yaml
customModes:
  - slug: my-mode                # /^[a-zA-Z0-9-]+$/
    name: My Mode
    roleDefinition: "..."        # required
    whenToUse: "..."
    description: "..."
    customInstructions: "..."
    groups: [read, edit, mcp, browser, command]
    source: project | global
    rulesFiles:                  # marketplace export only
      - relativePath: rules/x.md
        content: "..."
```
Defined at `packages/types/src/mode.ts:96-105` and extended at `packages/types/src/roomodes-schema.ts:31-39` [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/packages/types/src/mode.ts#L96-L107].

### Example real-world distribution

The shipped `DEFAULT_MODES` array at `packages/types/src/mode.ts:170+` includes `architect` and other built-in modes (`code`, `ask`, `debug`, `orchestrator`) — these are the schema reference. The architect mode entry illustrates how rich mode-specific instructions can be: a multi-paragraph `customInstructions` field that defines the mode's procedural behavior, plus `groups` like `["read", ["edit", { fileRegex: "\\.md$", description: "Markdown files only" }], "mcp"]` that gate tool access via regex [OFFICIAL-SOURCE — RooCodeInc/Roo-Code/blob/ad25634/packages/types/src/mode.ts#L168-L180].

A practice author targeting Roo can: (a) publish a mode YAML to the Marketplace and have users install via the in-extension Marketplace UI — `SimpleInstaller.installMode` at `src/services/marketplace/SimpleInstaller.ts:33-90` calls `customModesManager.importModeWithRules`, which atomically imports the mode definition and writes its `rulesFiles` payload into `.roo/rules-<slug>/*.md` — installing both the behavior and the rule corpus in one step; (b) publish a `.roomodes` file plus accompanying `.roo/rules-<slug>/*.md` files to a template repo for users to copy; or (c) target the cross-tool `.agents/skills/` directory which Roo will discover alongside `.roo/rules-*/` via `getAgentsDirectoriesForCwd` (`src/core/prompts/sections/custom-instructions.ts:368`).

Hooks: there is **no formal hook system** in Roo Code (a search for hook-named directories returns only test files and tree-sitter parsers, not lifecycle hooks). Behavior changes are achieved entirely via mode `customInstructions` and rule files. The `groups` field on a mode plus the optional `fileRegex` per-group is Roo's analog to a permission/tool gate, not a hook.

Multi-agent compatibility for Roo is the second-broadest in the sample after Cline: Roo reads `AGENTS.md`, `AGENT.md`, `AGENTS.local.md`, `.clinerules`, `.cursor/rules/`, `.cursorrules`, `.github/copilot-instructions.md`, `.roorules`, `.roorules-<mode>`, `.clinerules-<mode>`, plus its own `.roomodes` and `.roo/rules-*/`. The built-in `/init` command at `src/services/command/built-in-commands.ts:13-50` actively merges all of these sources into a single AGENTS.md as a "convergence" action — Roo treats AGENTS.md as the canonical multi-agent surface.

## 5. Cline — `cline/cline`

### Repo + version + SHA

`https://github.com/cline/cline` HEAD at clone time: `5fe6c9a8ce10ce2b7168ebe45ef91dc9e8a38464` (`Add Z AI GLM-5.1 model (#10409)`). VS Code package version per `package.json:5` is `"3.81.0"` [OFFICIAL-SOURCE — cline/cline/blob/5fe6c9a/package.json#L1-L10].

### Distribution mechanism

Cline is the only project in the five with a **genuine Claude Code-style hooks system**, plus the broadest cross-agent file ingestion.

**(1) `.clinerules` directory** — `src/core/storage/disk.ts:44-71` declares the canonical `GlobalFileNames` map. Key entries: `clineRules: ".clinerules"`, `workflows: ".clinerules/workflows"`, `hooksDir: ".clinerules/hooks"`, plus four skill-source directories: `clineruleSkillsDir: ".clinerules/skills"`, `clineSkillsDir: ".cline/skills"`, `claudeSkillsDir: ".claude/skills"`, `agentsSkillsDir: ".agents/skills"` [OFFICIAL-SOURCE — cline/cline/blob/5fe6c9a/src/core/storage/disk.ts#L44-L71]. The cline-rules loader at `src/core/context/instructions/user-instructions/cline-rules.ts:31-75` walks the `.clinerules/` directory recursively, parses YAML frontmatter via `parseYamlFrontmatter` for conditional activation, and merges with remote-config rules pulled from a Cline cloud endpoint [OFFICIAL-SOURCE — cline/cline/blob/5fe6c9a/src/core/context/instructions/user-instructions/cline-rules.ts#L21-L80].

**(2) Multi-source skills loader** — `src/core/storage/disk.ts:207-216`, `getSkillsDirectoriesForScan` returns six paths in priority order: `.clinerules/skills`, `.cline/skills`, **`.claude/skills`**, `.agents/skills` (all project), then global `clineSkillsDir` and `agentSkillsDir` [OFFICIAL-SOURCE — cline/cline/blob/5fe6c9a/src/core/storage/disk.ts#L199-L218]. **Cline reads Claude Code's `.claude/skills/` directly with no translation.** This is a load-bearing finding parallel to Continue's behavior.

**(3) Hooks system** — `src/core/hooks/hook-factory.ts:103-130` defines the `Hooks` interface with nine event types: `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `TaskStart`, `TaskResume`, `TaskCancel`, `TaskComplete`, `Notification`, `PreCompact` [OFFICIAL-SOURCE — cline/cline/blob/5fe6c9a/src/core/hooks/hook-factory.ts#L103-L132]. Hook discovery at `src/core/storage/disk.ts:533-557` walks three sources: a runtime `--hooks-dir` CLI flag, the global `~/<...>/Cline/hooks` dir, and per-workspace `.clinerules/hooks/` directories [OFFICIAL-SOURCE — cline/cline/blob/5fe6c9a/src/core/storage/disk.ts#L533-L575]. Hook output schema (`hook-executor.ts:30-50`) supports `cancel: boolean`, `contextModification: string`, `errorMessage: string`. The 30-second timeout and 50KB context-modification cap are enforced at `hook-factory.ts:27-30`. **This is the closest peer to Claude Code's hook system in the named-targets sample.**

### Schema

Rule files: any markdown in `.clinerules/`. Optional YAML frontmatter parsed by `src/core/context/instructions/user-instructions/frontmatter.ts:38-60` — fail-open semantics; if YAML fails, the body is treated as the whole file [OFFICIAL-SOURCE — cline/cline/blob/5fe6c9a/src/core/context/instructions/user-instructions/frontmatter.ts#L38-L60]. Frontmatter fields are open-ended `Record<string, unknown>` and used by the conditional-rules engine at `rule-conditionals.ts`.

Skills: discovered as directories containing `SKILL.md`. The schema is whatever Claude Code's Agent Skills format defines, which Cline accepts wholesale.

Hooks: executable scripts in `.clinerules/hooks/` named after their event type (e.g. `PreToolUse.sh`). Stdin receives JSON event payload; stdout returns JSON response matching the `HookOutput` protobuf.

### Example real-world distribution

A practice author targeting Cline distributes a directory tree:
```
.clinerules/
├── *.md                       # rule files (can have frontmatter)
├── workflows/                 # named workflows
├── hooks/                     # PreToolUse.sh, etc.
└── skills/                    # SKILL.md trees
.claude/skills/                # also read directly
.agents/skills/                # cross-tool convention
```
Distribution path: commit the directory tree to a template repo, users copy it into their workspace. Cline also supports a "remote rules" cloud-config endpoint (`cline-rules.ts:55-65`) for enterprise teams. There is no marketplace command analogous to Roo's.

**Multi-agent compatibility**: Cline reads at minimum five rule formats (`.clinerules`, `.cursor/rules/`, `.cursorrules`, `.windsurfrules`, `AGENTS.md`) and four skill directories (`.clinerules/skills`, `.cline/skills`, `.claude/skills`, `.agents/skills`). It is the most omnivorous consumer in the sample. The `external-rules.ts:18-50` toggles system gives the user per-rule on/off control across all imported formats, persisted via `controller.stateManager.setWorkspaceState`. Each external format gets its own toggle key (`localCursorRulesToggles`, `localWindsurfRulesToggles`, `localAgentsRulesToggles`), so a Cline user can selectively enable, say, the `.cursor/rules/` set while disabling `AGENTS.md`.

**Hook integration depth**: the hooks system is the only one in this sample that supports cancellation (`PreToolUseHookCancellationError.ts`), context modification (the hook output's `contextModification` field is appended into the model's prompt), and a process registry for multi-hook coordination (`HookProcessRegistry.ts`). Hook discovery is cached via `HookDiscoveryCache.ts`, and hooks have a maximum 30-second wall-clock timeout from `hook-factory.ts:27`. The protobuf-defined event types in `src/shared/proto/cline/hooks.ts` give external practice authors a stable JSON schema to target. **For a Momentum-style practice module that wants behavioral enforcement (e.g., "block any tool call that writes to `node_modules/`"), Cline is the only target in this sample where a third-party hook can implement that policy** — Aider, Cody, and Roo can only enforce via prompt instructions, which the model may ignore. Continue likewise has no hook surface.

**The `continuedev/rules` translator CLI** (`https://github.com/continuedev/rules`, HEAD `9d52589`) is the practical bridge across these formats. `internal/formats/formats.go:36-119` codifies the format map: `continue` → `.continue/rules/*.md`, `cursor` → `.cursor/rules/*.mdc`, `windsurf` → `.windsurf/rules/*.md`, `claude` → `CLAUDE.md`, `copilot` → `.github/instructions/*.instructions.md`, `codex`/`amp` → `AGENT.md`, `cline` → `.clinerules/*.md`, `cody` → `.sourcegraph/*.rule.md` [OFFICIAL-SOURCE — continuedev/rules/blob/9d52589/internal/formats/formats.go#L36-L119]. A practice author writes one `.rules/*.md` set, then `rules render <format>` emits the per-tool layout. This is currently the only general-purpose translator that targets all five named projects.

**For Momentum's strategy**: Continue and Cline already consume `.claude/skills/` SKILL.md files directly with no shim, which means a single canonical Momentum skills layout reaches three agents (Claude Code, Continue, Cline) without translation. Roo Code is reachable via `.agents/skills/` mirror or Marketplace publish. Cody requires `.sourcegraph/*.rule.md` flattening. Aider requires `CONVENTIONS.md` flattening with no skill granularity preserved. The `continuedev/rules` CLI is a viable build-step output target if Momentum standardizes on `.rules/*.md` as a publishing intermediate, but it does not preserve Skills' multi-file hierarchy — Skills (with sibling reference docs) are a strict superset of any single-file rule format.

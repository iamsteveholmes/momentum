Momentum — Skill & Agent Landscape An analysis of how every Momentum skill composes work · 2026-05-14

This document maps every skill in the Momentum plugin and shows — for the ones that spawn subagents — what they spawn, what each agent owns, and which knowledge sources each agent actually reads. It exists so that grooming the composable-specialist-agents feature stops being a guessing game.

ASkill landscape

Every Momentum skill, tagged by its primary structural role. Spawner = invokes subagents via the Agent tool. Orchestrator = sequences other skills/CLI calls; mostly no direct subagent spawns. Agent = the skill itself is the subagent body (context:fork, allowed-tools: Read). Writer = writes documents directly (no spawn). CLI = pure shell/state plumbing. Stub/Deprecated = not implemented or retired.

BPer-skill diagrams

One diagram per skill that spawns at least one subagent or invokes a downstream skill. Each diagram shows: the skill node at top; the agents or sub-skills it spawns; what each agent owns (the green "writes" line); what each agent reads (the purple "reads" line). Spawn pattern (parallel · sequential · single) is labelled below the skill.

sprint-dev opus · effort high · 6 roles · 2 waves

Executes the active sprint. Dev wave runs unblocked stories in parallel; post-merge AVFL runs once across the bundle; Team Review fans out code-review + qa + e2e + architect-guard.

Also invokes momentum:avfl once after all stories merge (Phase 4 — read-only gate). Reads sprints/index.json + stories/index.json + .feature specs for verifier prompts.

quick-fix opus · effort high · 5 phases · orchestrator + spawner

Single-story full-cycle workflow. Delegates story creation; spawns sole-writer agents for spec impact; selects a stack-specific dev via specialist-classify CLI; runs Team Review.

retro sonnet · effort high · auditor team + synthesizer

DuckDB transcript audit, then a 3-auditor fan-out (Phase 4), then a synthesizer writes the findings doc. Phase 5 routes Tier 1 findings to distill; Phase 6 writes sprint-summary.md and handoffs.

Phase 5 spawns momentum:distill per Tier 1 finding (inline subagent spawn). Phase 6 spawns momentum:feature-status (now deprecated → moving to canvas) and writes sprint-summary.md + handoff events to intake-queue.jsonl.

avfl opus · effort high · 4 sub-skill roles · model/effort fixed by benchmark

The Adversarial Validate-Fix Loop. Validators run in parallel; consolidator scores and dedupes; fixer applies. All four roles live as internal sub-skills, spawned via the Agent tool pointed at the sub-skill path.

Profiles control fan-out: gate = 1 enumerator/structural lens only · checkpoint = 1 per active lens · full = 8 total (1 enum + 1 adv per lens × 4 lenses) · scan = identical to full but no fix loop. Skepticism level passed to each validator; floor=2 enforced by benchmark.

distill sonnet · effort medium · discovery → tier-routed write

Practice-artifact distillation. Discovery pass classifies into Tier 1 (apply now) vs Tier 2 (create story); apply phase delegates accordingly. Orchestrator never writes; every change goes through a spawned writer.

assessment sonnet · effort high · roster agreed with developer

Produces an Assessment Record (ASR). Roster of discovery agents is proposed, the developer approves it, then they fan out in parallel. Findings are validated with the developer before they enter the ASR.

Step 5 writes the ASR to _bmad-output/planning-artifacts/assessments/aes-NNN-<slug>-<date>.md using references/asr-template.md. Step 6 updates the assessments index.md.

research opus · effort high · web + Gemini + AVFL corpus

Deep-research pipeline. Sub-question decomposition spawns N parallel research subagents; a single Opus synthesizer produces the briefing; Gemini Deep Research triangulates via cmux-browser; AVFL corpus mode validates the body of evidence.

feature-grooming sonnet · effort high · features.json maintenance

Bootstrap or refine the feature taxonomy. Two parallel discovery agents map FR clusters against existing capability clusters; the orchestrator synthesizes; the developer approves; features.json is written under a strict gate.

Orchestrator handles synthesis, value-analysis, developer interaction, and writing directly (no extra subagents). Writes features.json after approval gate; in bootstrap mode also writes aes-NNN-feature-value-gap.md and dec-NNN-feature-value-first.md.

feature-breakdown sonnet · effort high · enumerate stories per feature

Given a feature slug, produce the story enumeration that ships it end-to-end. Two perspectives in parallel (acceptance-first and value-gap-first); the orchestrator merges; triage routes the result.

refine sonnet · effort high · 2-wave planning artifact drift detection

Wave 1 detects drift between backlog and planning artifacts. If drift is found and the developer approves, Wave 2 fans out sole-writer agents that update the planning artifacts. Orchestrator never writes prd.md or architecture.md.

agent-guidelines opus · effort high · Gen-1 stack-rule generator

Discovers project stack via 4 scanner sub-skills, optionally researches breaking changes, then generates path-scoped rules and reference docs. The Gen-1 predecessor to constitution-builder.

Generation writes to .claude/rules/<tech>.md (Layer 1, path-scoped) and optionally docs/references/<tech>-patterns.md (Layer 2, reference). May also update CLAUDE.md with pointers. Templates: references/rule-template.md, references/reference-doc-template.md.

create-story sonnet · effort medium · delegates to bmad-create-story

The story file's owner is bmad-create-story (external BMAD skill). Momentum's create-story orchestrates around it: delegates the heavy lifting, injects a Momentum Implementation Guide section, records metadata, and runs an AVFL checkpoint over the result.

dev sonnet · effort medium · user-invocable: false · pure executor

Pure executor. Resolves the next unblocked story, creates a worktree, delegates implementation to bmad-dev-story, manages merge gate. Never reads .feature specs (Decision 30 black-box separation).

triage sonnet · effort high · 6-class delegator

Batch-classifies observations into six classes, then delegates by class. ARTIFACT → intake; DISTILL → distill; DECISION → decision (sequential, interactive). SHAPING/DEFER/REJECT → direct write to intake-queue.jsonl.

sprint-planning sonnet · effort high · story selection + activation

Curates the next sprint: reads master plan, picks unblocked stories, fleshes stubs, generates Gherkin specs, runs AVFL per story, activates. Per-story create-story spawns happen in sequence; AVFL checkpoint per story is delegated.

plan-audit sonnet · effort medium · gates ExitPlanMode

Audits the active plan for spec impact, classifies trivial vs substantive, creates a process story for substantive plans, runs an AVFL spec audit, and writes ## Spec Impact back into the plan file.

decision sonnet · effort high · SDR writer · optional research spawn

Walks an assessment or research source, captures adopt/reject/defer decisions per finding, writes a linked SDR document. Optional bmad-technical-research spawn if the developer requests deeper investigation.

impetus sonnet · effort low · top-level orchestrator

The voice of the practice. Reads the sanctum + sprint state; greets the developer; dispatches to whichever momentum:* skill the moment calls for. Owns no work artifact — owns the session-level orientation and dispatch.

EOpen questions

Each question pairs with "what would let us answer it." These are the cells where the discovery pass left genuine ambiguity.

Is dev-fixer a distinct role, or just dev spawned with a "fix this finding" prompt?
A decision. Sprint-dev Phase 4d uses sequential per-finding spawns and reuses the dev role contract. If we declare them unified, the dev-fixer-agent-definition story collapses into the dev base body.
Does dev-skills survive given the "skills are built with /skill-creator" direction?
Audit: when does momentum-tools specialist-classify return dev-skills? If only for SKILL.md / agent file changes, and those are now built by /skill-creator + constitution-builder, dev-skills becomes vestigial. The classification table itself (in quick-fix Phase 3) is the lookup to change.
Is there a "PM" agent? Quick-fix's orchestrator-purity comment lists it but no live spawn site appears to use it.
Grep + verification: search every workflow for an actual PM spawn. If none exists, remove the reference from the comment and the planned pm-base-body story.
Is retro's synthesizer a typed role or just an inline write-with-synthesis subagent?
A decision on whether documenter-agent-definition (story) creates a real plugin agent file or is closed as "no typed role — orchestrator spawns a generic writer when needed."
Distill's Enumerator and Adversary share names with AVFL's roles but specialize differently. One role with two prompts, or two roles with naming overlap?
A taxonomy decision. The pattern is "enumerator framing" vs "adversary framing" applied to different domains. If the framings are stable contracts, write base bodies; if each use site already specializes inline (as today), no plugin agent file is needed.
Do project state files (.momentum/stories/index.json, sprints/index.json, etc.) belong in the KB, or are they the canonical "not-in-KB" exception?
An explicit decision. State files are append-mutating live records; the KB is curated knowledge. The architecture seems to treat them as the only category that bypasses the KB. Confirming this in writing would let several Open Questions resolve cleanly.
Does code-reviewer get value from a project constitution, or is generic adversarial review enough?
Compare two trial runs of code-reviewer — one with the project's TDD/architecture rules injected, one without — on the same diff. If the deltas are material, code-reviewer joins frontend-dev-style per-project skills. If not, it stays a generic agent.
Could architect-writer (quick-fix) and "Architecture update agent" (refine wave 2) be unified? Same role contract: sole writer of architecture.md.
A naming + reuse decision. Same applies to prd-writer vs "PRD update agent." Sole-writer agents look like one role with two callers.
Are the 11 planned *-agent-definition stories (AVFL validator/enumerator/adversary/consolidator/fixer; retro auditors × 4 + documenter; code-reviewer; architect-guard; dev-fixer) the right unit of work?
Per-row verdicts from this document. The AVFL five already exist as sub-skills in avfl/sub-skills/. The retro three auditors and synthesizer are real roles. The dev-fixer question is above. Plugin code-reviewer and architect-guard agents exist but are stubs.
Is agent-guidelines superseded by constitution-builder + frontend-dev-style per-stack skills, or do its 4 scanners survive as a discovery primitive?
A direction. If skills like frontend-dev are the unit of project conditioning, agent-guidelines' generation step is replaced; its scanner sub-skills could still serve as standalone stack-inventory primitives or be folded into /skill-creator.
Does upstream-fix get built next, or get culled?
A decision. Today it's a stub deferred to "Story 4.X." Its conceptual home is the practice flywheel; the findings-ledger is in place. The question is whether the workflow is needed as a standalone skill or whether retro Phase 5 + distill already do this.

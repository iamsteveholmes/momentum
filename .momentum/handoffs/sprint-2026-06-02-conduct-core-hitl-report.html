01What conduct is, and what shipped

The problem it solves. Today, when Momentum builds a batch of work, the old builder interrupts you constantly — approve this, confirm that, resolve this conflict, fix-or-defer each issue. Roughly 17 stop-and-ask moments in a single build, almost all for things you'd always wave through. That constant interruption is the pain this sprint set out to kill.

The idea. conduct builds the entire batch on its own — writes it, reviews it, fixes what's wrong, merges it — and interrupts you exactly once, at the end, with a single report (this one). With one deliberate exception: for the rare change that's genuinely dangerous — could destroy data, rewrite history, or break everything downstream — it stops mid-build and asks, because those can't be safely undone. Everything else waits for the end.

What now exists that didn't before — five capabilities:

  1. The Conductor — a new engine that runs a whole build by itself: works out which items are ready, builds each in isolation, runs the quality checks, fixes what they find, merges each item when it passes, then hands you one report.
  2. A "know when to stop" safety valve — a shared rulebook plus a small engine that sorts every problem into routine (fix silently) vs high-stakes (security, irreversible, or far-reaching), and pauses to ask you only for the dangerous-and-urgent ones.
  3. A fixer — the builder's worker gained a "fix mode": hand it a problem and it either fixes it, or — if it's high-stakes — raises it for you instead of quietly fixing it.
  4. Real code-review tooling — the old placeholder reviewer was replaced with an actual adversarial bug-hunter, run without interruptions, and the other workflows were pointed at it.
  5. Rescoped helpers — the quality-checker now reviews one item at a time (so a problem is pinned to the item that caused it), and the builder's worker stopped managing git itself (the Conductor owns all of that now).

Important caveat — this is the spine, not the finished machine. What shipped is the core of those five capabilities and the safety machinery around them. Conduct is not yet a runnable replacement for the old builder: a few load-bearing pieces are deliberately left for a follow-up slice, and until they land, conduct can't run a build on its own and can't be adopted as the default. §06 lays out exactly what's live versus still hollow — read it before you decide, because it bears directly on what approving does.

How the build itself went — and why that's the real proof. conduct doesn't exist yet, so I ran this build by hand as the Conductor, following the model these 21 items define. Across 6 rounds, every item was built, independently reviewed, auto-fixed, and merged. The review layer caught ~93 problems and fixed them automatically; 14 of those were serious enough that, left unfixed, they'd have caused real harm — those are §03. The mid-build "stop and ask" alarm never once fired on a false alarm, and fired exactly once for a genuine high-stakes call (§04) — which is exactly the calibration the design was aiming for.

02What each piece is for

Plain-language purpose of all 21 work items — enough to understand why a problem in any of them would matter. The ones that needed correcting link down to their risk story in §03.

The blueprint

The design blueprintconduct-spec-revision-dec036

This story's job was to rewrite the conduct design spec so it actually matches the rule the team had just agreed to — a rule that loosened an earlier all-or-nothing stance. The earlier spec said two things absolutely: the human is asked to review only once, at the very end, and the system silently auto-fixes every problem it finds. The agreed-upon refinement keeps that as the default but carves out one narrow exception: a genuinely dangerous or build-breaking problem can interrupt the human mid-build. Because every other story in the build reads this spec as its blueprint, the guarantee here is that the blueprint tells the truth. Without this fix, every downstream builder would have faithfully re-implemented the old, stricter behavior — meaning a dangerous problem (say, a destructive action or a security gap) would have been quietly auto-handled or held silently until the end instead of being raised to a human in time.

Delivered: The conduct design spec now describes the agreed behavior end to end — a default single end-gate plus one narrowly-bounded mid-build escalation path, the new "raise it, don't silently fix it" handling for high-stakes findings, and a reconciliation note tracing each change back to the decisions it implements — so the rest of the build was constructed against a correct blueprint instead of an outdated one.

The safety vocabulary — how problems are classified & resolved

The shared finding rulebookdirected-fix-finding-schema  had a high-risk divergence → §03

This story wrote the single shared rulebook that describes what a "review finding" looks like — the common format every reviewer (and the auto-fixer) must use to report a problem, so that nothing one reviewer flags gets lost or misread by the next step. Without this one agreed-upon shape, each reviewer would speak its own dialect and the fixer could silently fix the wrong things, drop findings, or have no consistent way to tell a routine problem apart from a serious one that needs a human's eyes.

Delivered: There is now a single canonical finding-schema document that every reviewer and the fixer read, spelling out each field, how serious problems get raised instead of silently fixed, and exactly how each finding must be dispositioned (fixed, dismissed, triaged to the backlog, or escalated).
The risk-classification rulebookstakes-classification-rubric

This story wrote the single rulebook conduct uses to decide how serious any problem found during a build is — sorting every finding into one of three high-stakes buckets (security and access concerns; things that can't be safely undone like deletes, migrations, or production deploys; and changes whose failure spreads widely) or else "routine," and deciding the rare cases that must interrupt the build immediately versus the ones that wait for the final human review. Without one shared rulebook, the three different parts of the build pipeline that flag problems would each judge severity their own way, so a genuinely dangerous finding could be downgraded to "routine" and silently auto-fixed-and-shipped by one part while another would have escalated it — the exact inconsistency that lets serious risks slip past the human gate.

Delivered: There is now one authoritative, referenceable rubric document that defines every stakes class, the routine fall-through, and the deliberately narrow rule for when a finding interrupts the build mid-flight versus waiting for the expanded end-gate — written so the code-review adapter, the qa-reviewer, and AVFL all classify against it instead of each inventing their own classifier.
The fixer hand-off contractdirected-fix-invocation-contract

This story wrote the rulebook for how conduct's automatic fixer hands its results back to the orchestrator that runs the build: for every problem review finds, the fixer must report exactly what it did with it — fixed it, dismissed it with a reason, set it aside, or flagged it as too dangerous to silently fix and raised it for a human instead. Without this contract, a high-stakes problem (a database migration, a delete, a force-push, a security gap) could get quietly auto-fixed and buried alongside trivial ones, with no reliable way for the orchestrator to tell which findings were handled and which need human eyes.

Delivered: There is now a written contract of record defining the four ways a finding can be resolved, the rule that dangerous stakes-class findings are escalated to a human with the full what/why/evidence carried inline rather than auto-fixed, and the boundary that the fixer reports its dispositions but never pauses or prompts on its own.

The fixer — acting on what review finds

The fixer's fix-or-raise decisiondev-fix-mode-entry  had a high-risk divergence → §03

This piece teaches the builder's worker how to act on a problem that review found: quietly fix and commit ordinary problems on its own, but for high-stakes problems (anything touching security, anything destructive or irreversible like a migration or a force-push, or anything that reshapes the architecture) make no change and instead raise the issue to a human for a decision. The guarantee it gives the conduct system is that dangerous fixes never get applied silently and only land in front of the human when they genuinely warrant it — without this dividing line, either a risky change gets auto-committed and buried, or every trivial nit interrupts the human and the single end-of-build review turns into a firehose.

Delivered: The builder's worker can now take a directed problem report, sort it by how risky the fix is, auto-fix and commit the safe ones, and hand the dangerous ones back to a human as a self-contained escalation that says what the problem is, why it's risky, the evidence, and how urgent it is.
The automatic fix loopstage3-fix-loop-via-directed-dev  had a high-risk divergence → §03

This story builds the per-story fix loop: after a piece of work is reviewed, ordinary problems get fixed and re-checked automatically (up to three tries) with no human in the way, while a narrow class of genuinely high-stakes problems (security holes, irreversible changes like data migrations or force-pushes) are pulled out of the auto-fix path and raised to a human instead of being quietly changed. The guarantee it gives the conduct system is that routine issues fix themselves cheaply while dangerous ones can never be silently auto-corrected and slipped past; without this piece, an auto-fixer could spin forever, ship work it never actually repaired, or silently fix a security or destructive change a human should have seen.

Delivered: The system now has a working per-story fix loop that auto-resolves ordinary findings within a bounded three-try retry, marks genuinely stuck work as blocked instead of shipping it, and routes high-stakes findings onto a separate escalation channel that surfaces them to a human rather than fixing them silently.

The Conductor — the engine that runs a whole build

The Conductor skeleton + spineconduct-skill-scaffold-and-spine  had a high-risk divergence → §03

This story builds the backbone of the whole autonomous build system: the top-level skill that runs an entire sprint's worth of stories end to end and the ordered script of phases it follows. Its guarantee is that one and only one place is in charge of changing the codebase and launching helper agents, that the phases always run in the right order, and that the developer is asked nothing during a normal build until a single review at the very end. Without this piece there is no orchestrator at all — nothing sequences the build, no one owns the working tree, and every other part of conduct (the per-story builds, the auto-fix loop, the escalation path, the final report) has nothing to attach to.

Delivered: A working orchestrator skill now exists that drives a full sprint through pre-flight, per-story builds, post-merge review, end-to-end validation, and a single closing review — with all code-changing and agent-launching authority concentrated in one accountable place and no interruptions to the developer along the way.
The build schedulerconduct-build-phase-frontier  had a high-risk divergence → §03

This story builds the engine that runs an entire sprint's work autonomously: it figures out which work items have all their prerequisites met, launches all of them at once with no ceiling, and keeps the run moving as each item finishes, so the whole sprint builds in one continuous flow with a single human checkpoint at the very end. It also carries a narrow safety valve: if a work item hits a genuinely serious, can't-wait problem (something irreversible-and-about-to-happen, or something that breaks the whole build), that one branch pauses and raises it to a human instead of quietly pushing on. Without this piece, a single failed item would strand the rest of the sprint, the run would stall and need babysitting, and a serious problem could slip through unnoticed.

Delivered: The Conductor can now drive a full sprint from ready to merged on its own: it launches all unblocked work concurrently, relaunches newly-unblocked work the moment something merges, survives a failed item without halting the whole run, and pauses-and-escalates only the rare problems that clear a deliberately narrow seriousness bar.
The stop-and-ask engineconduct-stakes-timing-escalation-mechanism  had a high-risk divergence → §03

This piece is the one and only place in conduct that is allowed to stop an autonomous build mid-run and ask the developer a question — but only for a tiny, dangerous class of moments: an action that is about to do something irreversible (a destructive database migration, a delete, a force-push, a production deploy) or that would break the build itself. Everything else stays silent and gets collapsed into the single end-of-run report. Without this piece, conduct would have only two bad options: interrupt the developer constantly (the "firehose" it was designed to avoid) or never interrupt at all — meaning a destructive, irreversible action could fire automatically with no chance for the developer to stop it first.

Delivered: Conduct now has a single shared "pause and ask" mechanism that interrupts the build exactly once for each genuinely irreversible-and-imminent or build-breaking finding — presenting what changed, why it matters, and the evidence inline — lets the developer answer proceed / change / abort-that-branch, and resumes; everything below that bar continues to run untouched.
Per-story merge + conflict handlingconduct-merge-and-conflict-resolution  had a high-risk divergence → §03

This piece owns how each finished work item gets folded into the combined result: it rebases and merges every story, quietly fixes the conflicts it safely can, sets aside the ones it can't (preserving the work so nothing is lost), and — critically — stops and pulls the developer in only when fixing a conflict would touch something dangerous (a security/login boundary, an irreversible action like a delete or a database migration) or break the build. Without it, a single bad merge would either halt the whole run, silently discard a story's work, or — worst of all — silently rewrite something dangerous and ship it with nobody ever being asked.

Delivered: The conduct system can now integrate every story autonomously — auto-fixing safe conflicts, quarantining unmergeable ones without stopping the sprint, and raising a mid-flight alarm to the developer (and only for) the narrow set of conflict fixes that would alter a security boundary, do something irreversible, or break the build.
Tamper check on the test contractconduct-contract-freeze-check  had a high-risk divergence → §03

This piece guarantees that when conduct checks whether a story passed or failed, it checks against the exact same test criteria that were agreed on and locked in at the start of the build — not a version that was edited, regenerated, or quietly drifted in between. Without it, a story could "pass" against criteria nobody approved, or "fail" against criteria it was never given, and the final report the developer trusts would be built on a result that was silently scored against the wrong yardstick.

Delivered: Conduct now takes a fingerprint of each story's test criteria when work begins, re-checks that fingerprint right before scoring the story, and refuses to record a result for any story whose criteria changed underneath it — handling the mismatch itself without ever interrupting the developer.
Skip-redundant-QA routingconduct-coverage-disposition-branch  had a high-risk divergence → §03

This piece decides, for each work item the autonomous builder processes, whether that item gets its own quality-check run during the build or whether its check is deliberately postponed to a later, combined integration test that already covers it. The guarantee it gives is that postponing a check only changes when and where the check happens — never whether it happens — so without this piece a work item could be quietly waved through with no verification at all, or be checked twice and produce conflicting signals.

Delivered: The builder now reads each item's frozen "how should this be verified" setting and routes accordingly — running a dedicated check, or recording a skip that names the exact later test responsible for verifying it — and falls back to running a full check whenever that setting is missing, unrecognized, or names no downstream owner.
Can-we-even-start guardsconduct-preflight-halts  had a high-risk divergence → §03

This story is the front door of every conduct build: before any work starts, the system runs a handful of "can we even begin?" checks (no sprint, sprint not turned on, missing approvals, leftover broken state) and then quietly cleans up the workspace from any prior interrupted run so the build starts from a known-good state. The guarantee it gives is that a build only ever starts from a validated, clean baseline, and once running it never stops to pester the developer — with one narrow, deliberate exception for genuinely dangerous moments. Without this piece, a build could either start on top of broken leftover state or constantly interrupt the developer, defeating the whole point of an unattended build.

Delivered: The conduct system now has a working start-up phase that refuses to begin when something is genuinely wrong, automatically resets and re-runs any story left half-finished by a prior interrupted session without asking the developer anything, and then hands off to the build from a clean baseline.

The code-review tooling

The non-interactive review drivercode-review-adapter-noninteractive-driver

This story builds the thin connector that lets conduct run its automated code reviewer on a single story's changes completely hands-off — it forces the reviewer to run start-to-finish, report what it finds, and never stop to ask a person a question or edit any files itself. Without it, the reviewer would freeze the whole autonomous build the first time it hit its usual "pause and confirm" prompt, or worse, start rewriting files behind the builder's back.

Delivered: conduct can now run adversarial code review on one story's changes unattended — it returns the full list of findings (or a clear "nothing found"), touches no files, and stops early with a plain, fix-it message if the reviewer's required config is missing instead of failing somewhere deep inside.
Translating review output into the rulebookcode-review-adapter-normalize-triage  had a high-risk divergence → §03

This piece is the translator that takes the output of the code reviewer and reshapes it into the one common "finding" format that conduct's auto-fixer understands, tagging each finding with how high the stakes are so the system knows which problems can be fixed silently and which deserve a human's attention. Without it, the fixer would either be locked to one specific review tool's private vocabulary or would receive findings with no stakes signal at all, which would quietly defeat the whole safety mechanism that decides when to escalate to a person.

Delivered: conduct now has a working adapter that converts the code reviewer's tool-specific result buckets into the system's single canonical finding format, drops only the reviewer's own noise, and stamps every surviving finding with a stakes class drawn from the shared rubric.
Retiring the placeholder reviewercode-review-adapter-retire-stub

This story's job was to make sure there is exactly one real code reviewer in the build, by retiring an old placeholder reviewer and pointing the review command at the genuine review engine. The guarantee it gives conduct is that every time the build asks for a code review, it gets actual review findings in the shape the rest of the system understands — never a hollow placeholder. Without this, the build could route to the dead stub, believe a change had been reviewed when it had only received empty placeholder text, and ship unexamined code.

Delivered: Invoking the code-review command now always drives the real adversarial review engine and returns properly shaped findings, while the old placeholder reviewer is gone and can no longer be reached.
Pointing sprint-dev at the real reviewercode-review-adapter-repoint-sprint-dev  had a high-risk divergence → §03

This story rewires the sprint-dev workflow's per-story code review so it goes through conduct's single shared review adapter instead of a retired, dead review path — guaranteeing that every story's changes are reviewed by the same reviewer, producing findings in the one consistent shape the rest of conduct (triage, auto-fix, escalation, the human end-gate) is built to read. Without this piece, sprint-dev would either call a reviewer that no longer exists (breaking the run) or emit findings in a stale, incompatible shape that downstream stages can't process — quietly splitting the practice into two divergent review paths.

Delivered: Sprint-dev now sends each story's real code changes to the shared review adapter and gets back normalized findings, with no remaining reference to the old retired reviewer.
Pointing quick-fix at the real reviewercode-review-adapter-repoint-quick-fix  had a high-risk divergence → §03

This story rewires the "quick-fix" flow — the fastest single-fix path a developer reaches for — so its code-review step calls the current, canonical review engine instead of an old retired one. The guarantee it gives conduct is that even the speedy single-fix path gets the same real, normalized code review every other path gets; without it, quick-fix would point at a review that no longer exists, and could appear to pass while no review actually happened.

Delivered: The quick-fix flow now runs its code review through the live review engine and returns the same normalized findings as every other Momentum review, with no leftover reference to the old retired path.

The helpers, rescoped

Per-story quality checkingqa-reviewer-rescope-per-story-contract

This piece is the inspector that checks each story's work in isolation, before it merges: it runs that story's own tests (never just reads the code and assumes it works), marks every requirement as met, partly met, missing, or unverifiable, and — critically — labels each problem it finds with how dangerous it is (touches security, touches an irreversible operation like a database migration or deletion, or is ordinary). It gives conduct the guarantee that no story is declared "done" on the strength of source code that merely looks right, and that the build's later safety gates receive an honest danger signal stamped at the moment the problem is discovered. Without it, broken work could pass review unrun, and a serious problem sitting on a destructive operation could be mistaken for a harmless one and quietly auto-fixed.

Delivered: conduct now has a per-story reviewer that verifies one story's changes against that story's own contract by actually executing its tests, and stamps each finding with a danger class at the source — replacing the old whole-sprint, after-the-fact review.
What the builder is allowed to readdev-read-contract-part-a-header

This story teaches the automated coding agent to read the plain-language "acceptance target" written at the top of each work item's verification contract and to check its own work against that target before declaring the item finished — while strictly forbidding it from reading the full verifier (so it can't game the test) or from deciding how its work gets graded. Without this, the coder and the later grader could be aiming at different targets, letting work that misses the real acceptance bar slip through as "done."

Delivered: The coding agent now reads the contract's plain-language self-check header, confirms its work against it before signaling done, and records that the self-check happened — and it falls back cleanly to the story's written acceptance criteria when no such header exists.
Taking git away from the builderdev-strip-merge-cleanup-authority

This story stripped the per-story developer agent of any power to merge code, clean up its own working directory, manage build locks, or ask a human how to recover from a crash — leaving it as a pure code-writer that only reports what it changed. The guarantee is that exactly one component (the Conductor) owns all git history changes and the single human approval gate; without it, every story-level agent could independently rewrite git history, leave stale locks behind, or pop up its own ad-hoc recovery prompts outside the one intended checkpoint.

Delivered: The developer agent now ends each run by reporting "implementation complete" plus the list of files it touched and nothing more, with all merge, worktree-cleanup, lock-handling, and crash-recovery authority removed and handed to the Conductor.

03Where the build diverged — the high-risk moments

These are the 14 divergences that mattered: places where the first attempt got something wrong that, unfixed, would have caused real harm. Each was caught by the build's own review before anything merged, fixed, and re-verified. Scan the headlines; open any one for the full story — what the piece does, why its guarantee was written the way it was, where reality diverged, the risk that created, and why the resolution is acceptable.

▶A story that the auto-fixer gave up on was getting merged into the shared build anyway, so unrepaired, still-failing work would have shipped looking just like work that passed.
The automatic fix loop · stage3-fix-loop-via-directed-dev

What this part of conduct does: this is the loop that, after a piece of work is reviewed, automatically fixes the problems review found, retrying a bounded number of times and then giving up. It is the most autonomous, least-watched stretch of the entire build, where the system decides on its own whether a story is good enough to fold into the shared result.

Why we wrote a guarantee around it, and why that way: the signature failure of any auto-fixer is declaring victory it did not earn, treating a story whose problems were never actually resolved as finished. So the contract pinned two non-negotiables: the loop retries at most three times and then stops (it can never spin forever), and a story still failing after those three tries is LEFT OUT of the merged result. That second rule is the important one, because silently merging unrepaired work is the worst thing an autonomous builder can do, hiding a defect inside a result that otherwise looks clean.

Where reality diverged: the first build honored the three-try limit and correctly recorded the still-failing story as blocked, and then merged it into the shared sprint branch anyway. The author had fused two separate ideas: do not halt the whole build (correct, keep building the other stories) with merge this one broken story (wrong). The two are independent, and only the first was supposed to be true.

The risk that created, and what catching it removed: unfixed, a story with failing tests or unmet acceptance criteria would have landed in the integrated build indistinguishable from one that genuinely passed, a silent defective merge, the exact outcome the guarantee existed to prevent. The adversarial review caught it before merge. The fix now leaves a blocked story out of the merge entirely, files it as a follow-up so the work is not lost, and keeps building the remaining stories.

Why the outcome is acceptable: the corrected behavior matches the property the contract was protecting, broken work does not ship and the build still does not stall on the other stories, and it was re-checked after the fix, so the safeguard is actually enforced rather than merely intended. A second high-severity slip in the same branch, the blocked story being set to a status the underlying tool rejects (which would have left it stuck rather than cleanly closed), was caught and corrected the same way.

▶The safety alarm that was supposed to stop the system before silently rewriting a security boundary or breaking the build was wired up so that nothing ever pulled it — it would never have fired.
Per-story merge + conflict handling · conduct-merge-and-conflict-resolution

What this part of conduct does: when two stories edit the same code in conflicting ways, the system tries to reconcile them on its own. Most of the time that is fine and should happen silently. But some reconciliations are dangerous — the only way to make the code merge is to change how login or permissions work, to perform something that can't be undone, or to make a change that breaks the build for every story behind it. For exactly those cases, this piece is supposed to stop, hand the situation to the developer, and record it as "raised to a human," never as a quiet auto-fix.

Why we wrote a guarantee around it, and why that way: the governing decision for conduct was deliberately aggressive about staying out of the developer's way — by default, resolve conflicts silently and keep moving. That default is right for the ninety-nine ordinary cases and dangerous for the hundredth, because the same silence that keeps the build flowing would also let the system quietly redraw a security boundary or break the build with no one watching. So the contract carved out one narrow, non-negotiable exception: a conflict fix that touches a high-stakes area OR breaks the build must leave the silent path and be raised to the developer mid-flight. The whole point of the story was to make that alarm real — a wire the system actually pulls, not a promise written in the margins.

Where reality diverged: the alarm was built as a sign on the wall, not a switch. The escalation step existed in the document, but it was written as a passive note that no part of the running flow ever reached — the conflict-resolution path routed straight past it to "apply the fix and move on." On top of that, the rule describing when to raise the alarm used "high-stakes AND build-breaking" where the contract demanded "high-stakes OR build-breaking," so even if something had reached the alarm, a fix that merely broke the build would have slipped under the bar.

The risk that created, and what catching it removed: left as written, the single most dangerous thing this system can do would have happened invisibly. A conflict fix that rewrote an authentication boundary, deleted something irreversibly, or broke the build for every downstream story would have been applied automatically, recorded as an ordinary silent auto-fix, and the developer would never have been asked — the exact catastrophe the guarantee was created to prevent. The adversarial review caught that the alarm was dead before any of it shipped; the fix turned the note into an actual executed step that the resolution path now routes through, and corrected the bar to "OR" so a build-breaking fix raises the alarm on its own.

Why the outcome is acceptable: the resolved behavior now does what the contract was protecting — dangerous and build-breaking conflict fixes stop and surface to the developer as genuine escalations, while ordinary conflicts still resolve silently and unmergeable ones are quarantined without halting the run — and it was re-verified after the fix, so the safety alarm is one that actually fires, not just one that is described.

▶A security or destructive problem the reviewer had already dismissed as a false alarm would still have been raised to the human as if it were real — turning the one human review into noise.
The fixer's fix-or-raise decision · dev-fix-mode-entry

What this part of conduct does: this is the moment, deep in the autonomous build, where the worker decides what to do with a problem review found. Ordinary problems it fixes and commits on its own; genuinely dangerous ones — anything touching security, anything destructive or irreversible, anything that reshapes the architecture — it refuses to touch and instead raises to a human as an escalation. It is the one valve that lets routine work flow fast and silent while peeling off only the high-stakes problems for human eyes.

Why we wrote a guarantee around it, and why that way: the entire promise of conduct is a single human review at the end of the build, not a stream of interruptions. So the escalation valve has to be precise in both directions. It must catch the genuinely dangerous problems — those can never be silently auto-fixed and buried — but it must be just as disciplined about what it does NOT raise. Review itself can flag something and then conclude on closer inspection that it was a false alarm; the rules say a false alarm should be quietly set aside with a written reason, never pushed in front of the human. The contract pinned this down explicitly: a problem gets escalated only when it is both high-stakes AND a real finding.

Where reality diverged: the first build got the high-stakes detection right but escalated those problems unconditionally — even ones the reviewer had already marked a false alarm. The spec went so far as to say it escalates "regardless" of whether the finding was real. The author had collapsed two questions into one: "is this high-stakes?" got answered, but "is this actually a real problem?" got skipped on the dangerous branch entirely.

The risk that created, and what catching it removed: unfixed, every false alarm that happened to look security-related or destructive would have been pushed to the human end-gate as if it were a live danger — exactly the noise the single-gate design exists to eliminate, and a direct contradiction of the story's own rule that only real findings get escalated. Worse, it would erode trust in the gate itself: a human who learns the dangerous-problem queue is full of false alarms stops reading it carefully. The adversarial review caught it before merge; the fix splits the dangerous branch so a real high-stakes problem is escalated and a false alarm is quietly dismissed with a written reason.

Why the outcome is acceptable: the resolved behavior now matches the guarantee the contract was protecting — the human gate sees real dangers and nothing else, and false alarms are set aside cleanly with a recorded reason — and it was re-checked after the fix and came back clean, so the valve is actually precise, not just intended to be.

▶The emergency "stop and ask before doing something irreversible" alarm was wired to a trigger signal that nothing in the system ever sends — so it could never have fired.
The stop-and-ask engine · conduct-stakes-timing-escalation-mechanism

What this part of conduct does: it is the single safety valve that lets an otherwise hands-off, fully autonomous build stop itself and ask the developer before it does something it can't take back — a destructive migration, a delete, a force-push, a production deploy — or before it commits to something that would break the whole build. It is the only sanctioned mid-build interruption in the entire system.

Why we wrote a guarantee around it, and why that way: the danger with an autonomous builder is asymmetric. Interrupt too often and you re-create the constant-nagging "firehose" the system was explicitly built to escape; interrupt too rarely — or never — and an irreversible action slips past with no human in the loop. So the contract pinned the alarm to fire on one narrow, high bar and nothing else: irreversible-and-imminent OR build-invalidating. The whole point of the guarantee was that this alarm reliably fires on exactly those moments and stays silent on everything else.

Where reality diverged: the engine was built to consume a shared "finding" record that the rest of the system produces — it is supposed to read that record, not invent its own fields. But the build quietly added a brand-new required field of its own (an "escalation reason" tag) and wired the alarm to fire only when that tag was present and set to one of two values. Nothing upstream — none of the reviewers, validators, or fixers that actually produce findings — knows that field exists or ever sets it. The upstream schema already carries the danger signal a different way (inside an existing "timing" marker), and the engine simply hadn't been pointed at it.

The risk that created, and what catching it removed: because no producer ever sets the tag the alarm was listening for, the condition could never be true — the mid-build pause could never fire. The one safety mechanism whose entire reason for existing is to stop irreversible actions would have sat there as dead code: a destructive migration or force-push would have rolled through autonomously with the developer never asked, while the report still claimed an escalation tier existed. The adversarial review caught it before merge by searching the whole codebase and finding the trigger field referenced nowhere but in this story's own two files.

Why the outcome is acceptable: the fix removed the invented field entirely and re-keyed the alarm onto the existing upstream signal the schema already guarantees — so the engine is now a true consumer of the shared record, and the bar fires on exactly the irreversible-and-imminent or build-breaking findings the guarantee was written to protect. It was re-verified after the fix, so the safety valve is now actually wired to something that fires, not just intended to.

▶A finding that was valid but belonged to a different piece of work had no way to be set aside — the rules would have forced the system to auto-fix code that wasn't part of this job.

What this part of conduct does: this is the rulebook that decides, for every problem a reviewer finds, what happens to it — get it fixed, dismiss it with a reason, raise it to a human, or set it aside as "real, but it belongs to a different piece of work." That last bucket exists so a reviewer who spots something legitimately broken but outside the current job can route it to the backlog instead of forcing it into the work in front of them.

Why we wrote a guarantee around it, and why that way: this schema is the keystone — every reviewer and the auto-fixer inherit its rules, so any inconsistency here multiplies across the whole build. The contract therefore demanded that the routing rules be internally self-consistent: every outcome the schema names as possible must actually be reachable, and a finding must always land in the bucket that fits it. A dead-end rule here isn't a typo — it silently mis-routes work everywhere downstream.

Where reality diverged: the schema named the "set it aside / send to the backlog" outcome, but its own processing rules made that outcome impossible to ever reach. The risk that created, and what catching it removed: unfixed, a reviewer flagging a genuine problem that belonged to a different component would have had it auto-fixed inside the current job — the system silently editing code that wasn't part of this work, instead of parking it on the backlog where it belonged. The adversarial review caught the contradiction before merge; the fix added the missing rule so legitimate out-of-scope findings now route correctly to the backlog, and narrowed the old rule so it no longer overlaps.

Why the outcome is acceptable: the resolved rules now make every named outcome actually reachable and route each finding to the bucket the contract intended — the "set aside to backlog" path works, auto-fix only touches in-scope work — and the document was re-reviewed after the fix, so the schema's self-consistency guarantee is genuinely enforced rather than merely claimed.

▶The build's only mid-run safety alarm was wired so that the most likely kind of serious problem, one that breaks the build outright, could never set it off.
The build scheduler · conduct-build-phase-frontier

What this part of conduct does: this is the autonomous engine that runs a whole sprint with no human in the middle, plus the one safety valve allowed to interrupt that flow mid-run. As each work item finishes, its result comes back to the engine as one of two signals: it either merged successfully or it failed. The safety valve checks, before letting the run continue, whether that item raised a serious-enough problem to deserve a human's eyes right now.

Why we wrote a guarantee around it, and why that way: an autonomous builder with a human gate only at the very end has exactly one weak spot, a problem severe enough that waiting until the end would be too late. The contract deliberately drew that line extremely narrow: only two kinds of problem may interrupt the run mid-flow, something irreversible that is about to happen, or something that invalidates the build itself. The guarantee was written this way precisely so those two categories can never be quietly swept onto the auto-fix path; for them, the run must pause that branch and raise the alarm to a human.

Where reality diverged: the alarm was connected to only one of the two ways a work item can report back. It listened on the merged signal but not on the failed signal. The trouble is that one of the two named trigger conditions, a problem that breaks the build, is by its nature the kind that stops an item from ever reaching merged. It surfaces as a failure instead. So the single category most likely to be genuinely build-breaking arrived on the one channel the alarm wasn't watching; the failed item just got retried a couple of times and then quietly set aside.

The risk that created, and what catching it removed: unfixed, the most dangerous class of problem the safety valve exists to catch, a build-invalidating finding, could never actually trigger it. That is a safety mechanism that looks present but would not fire when it mattered most. The adversarial review caught it before merge; the fix wires the same alarm onto the failed channel too, so a serious problem raised through a pipeline failure now reaches a human exactly as one raised through a successful merge would.

Why the outcome is acceptable: the resolved behavior now matches the guarantee the contract was protecting, both named trigger conditions can actually pause-and-escalate regardless of which way the work item reported back, and it was re-verified after the fix, so the safety valve is genuinely connected on every path, not just intended to be.

▶The integrity guard that proves each story was scored against its agreed-on test criteria was written but never actually wired in, so it would not have run before any story's result was recorded.

Why we wrote a guarantee around it, and why that way: conduct's entire promise is that the developer can walk away during a build and come back to one trustworthy report. That report is only as honest as the individual pass/fail results it rests on. So the contract pinned the guard to a specific moment and cadence: it must run as the *first step of each story's verification*, once per story, against that story's own locked fingerprint. The phrasing was deliberate — a single check at one point in the run would be worthless, because a contract could drift for any one story at any time. The guarantee had to fire story-by-story, right before each score is taken, or it guarantees nothing.

Where reality diverged: the guard was fully written — it knew how to re-fingerprint, compare, and stop on a mismatch — but it was left sitting in the workflow with no one calling it. The risk that created, and what catching it removed: unwired, the guard would never have stood between a drifted contract and a recorded result. Why the outcome is acceptable: the resolved behavior matches what the contract was protecting — the guard now runs per story, immediately before each result is taken, and blocks scoring against drifted criteria — and the build re-verified that the check actually fires in that position, so the integrity property is enforced in practice, not just written down.

▶The translator was putting the wrong kind of label into the slot reserved for the reviewer's pass/fail judgment, so findings would have entered the system silently malformed instead of in the uniform shape every other part depends on.

Why we wrote a guarantee around it, and why that way: the entire reason this story exists is uniformity. Why the outcome is acceptable: the resolved behavior now matches the guarantee the contract was protecting — every forwarded finding carries the correct verdict in the correct slot and conforms to the one canonical shape — and it was re-verified after the fix, so uniformity is actually enforced rather than merely intended. A related over-reach found in the same pass (a class of low-priority findings that would have been silently auto-fixed against the adapter's stated intent) was corrected at the same time.

▶A bug made per-story code review look at no code at all — every story would have passed review without a single line actually being checked.
Pointing sprint-dev at the real reviewer · code-review-adapter-repoint-sprint-dev

What this part of conduct does: before a sprint's work is finalized, conduct reviews each story's changes one at a time. To do that, it first has to carve out exactly the set of changes a given story introduced and hand that slice to the reviewer. The reviewer is deliberately trusting — it reviews whatever changes it's given, and does not go digging through the project's history to second-guess whether it received the right slice. So this step's one job is to correctly isolate a single story's changes out of a branch where many stories have already piled up together.

Why we wrote a guarantee around it, and why that way: because the reviewer trusts its input completely, the input has to be right — a wrong slice means the wrong code gets reviewed, with nothing downstream to catch the mistake. So the contract didn't just ask "did review run?"; it demanded that the per-story review produce *genuine* findings on the story's *true* changes — the same findings you'd get by reviewing that exact change directly — and explicitly treated an empty or placeholder result as a failure. The guarantee was framed that way precisely to close the most dangerous loophole: a review step that appears to run and pass while having actually inspected nothing.

Where reality diverged: the code that carved out each story's changes computed the wrong slice — twice. The first version counted commits backward from the tip of the shared branch, which only ever landed on the right boundary for the single most-recently-merged story; every earlier story got a truncated, empty, or polluted set of changes. A first fix swapped in a different range that looked correct but, given how sprint-dev actually stitches branches together (it fast-forwards them into one straight line), silently resolved to an *empty* slice for every story. So at review time, each story handed the reviewer nothing.

The risk that created, and what catching it removed: an empty slice means the reviewer finds nothing — and "nothing found" reads exactly like "clean pass." Every story would have sailed through its dedicated code-review leg without a single line ever being examined, and the run would have reported success. That is the precise failure this story existed to prevent: review that passes because review never actually happened, with real defects in the code escaping entirely and invisibly. The adversarial review caught it before it shipped — including catching that the *first* fix had quietly reintroduced the same emptiness — and the corrected version captures the branch position immediately before and after each story merges, so the slice always reflects exactly what that story changed.

Why the outcome is acceptable: the resolved behavior now matches the guarantee the contract was protecting — each story's review runs on that story's real changes and yields the same normalized findings a direct review would, never an empty pass — and it was re-verified after the refix (reproduced in a scratch repository mirroring sprint-dev's exact merge sequence) to confirm the slice is correct rather than merely intended.

▶Quick-fix would have reviewed only the last commit of a multi-commit fix, so most of the changed code could ship un-reviewed while the review still reported "clean."
Pointing quick-fix at the real reviewer · code-review-adapter-repoint-quick-fix

What this part of conduct does: this is the code-review step inside quick-fix — the streamlined, single-fix flow a developer uses when they want the fastest turnaround. When the fix is done, this step hands the code that changed to the review engine so problems get caught before the fix is merged.

Why we wrote a guarantee around it, and why that way: the whole point of repointing this step at the real review engine is so quick-fix gets a genuine review, not a hollow one. The story's own description named the exact nightmare to avoid: a review that surfaces as "review passed" when the review never actually ran. So the contract insisted the step feed the review engine the story's real changes and return real, normalized findings — not a partial or placeholder result that merely looks like a clean pass.

Where reality diverged: the step built the bundle of code to review as "the difference introduced by the last commit." But quick-fix merges a fix by replaying its commits onto the main line one after another, with no single combining commit. The dev agent routinely makes several commits per fix. So "the last commit" was literally just that — the final commit — and every earlier commit's changes were quietly left out of what got reviewed.

The risk that created, and what catching it removed: had this shipped, a multi-commit fix would have been reviewed against only a sliver of its actual changes. The review engine, seeing almost nothing, could legitimately report no problems — handing the developer a confident "clean" while the bulk of the new code was never looked at. That is precisely the silent false-pass the guarantee existed to prevent, and it is most dangerous here because quick-fix is the path people trust to be quick and done. Why the outcome is acceptable: the resolved step now reviews the fix's complete set of changes, which is exactly the real-review guarantee the contract was protecting, and it was re-checked clean after the fix — so quick-fix's review is now actually thorough, not just nominally present.

▶A work item could be marked "someone else will verify this later" while naming no one — so its quality check would be silently skipped and never picked up by anyone.
Skip-redundant-QA routing · conduct-coverage-disposition-branch

What this part of conduct does: for every work item the autonomous builder handles, this is the fork in the road that decides how the item gets verified — either it earns its own dedicated quality-check run right then during the build, or that run is deliberately skipped because a later, combined integration test is already on the hook to verify it. When it takes the "skip now, verify later" path, it is supposed to leave behind a clear note recording the skip and naming the exact downstream test that will pick the item back up.

Why we wrote a guarantee around it, and why that way: the entire reason this fork exists is to save redundant work without ever losing coverage — so the one outcome it must never produce is a work item that gets skipped now and verified by no one ever. We built a deliberate safety net into the contract for exactly this: if the verification instructions are missing, unrecognized, or otherwise malformed, the builder must default to running the full check rather than skipping it. Where reality diverged: the safety net was only watching half the data. It checked that the verification *mode* was a recognized value, but it never checked that the named downstream owner was actually filled in — and the data format explicitly allows that owner field to be left empty. So a work item could be validly marked "skip now, verify later" while naming no later test at all, and the builder would honor the skip, drop the build-time check, and move on. The risk that created, and what catching it removed: unfixed, a malformed assignment — one that said "defer" but named no one — would have caused that work item's quality check to vanish entirely: skipped during the build, owned by no downstream test, never flagged. It would have shipped looking exactly like a fully verified item while in fact never being verified at all, which is the precise silent coverage gap this whole fork was built to prevent. The adversarial review caught it before merge; the fix extends the safety net to also require a real named owner, and when one is missing the builder now refuses to skip and runs the full check instead.

Why the outcome is acceptable: the resolved behavior now matches the guarantee the contract was protecting — a work item can only be skipped when there is genuinely someone downstream to catch it, and any malformed or incomplete instruction falls back to verifying, never to silently dropping the check — and the fix was re-verified after the change, so the no-silent-skip safety property is actually enforced, not merely intended.

▶The start-up check used a wrong, hand-typed list of valid story states, so a perfectly normal leftover story could trip a false "cannot start" alarm and refuse to launch a build that should have run cleanly.

What this part of conduct does: before the build begins, one of the start-up guards inspects every story's recorded state and stops the whole run if it finds anything it doesn't recognize — the safety net that refuses to build on top of corrupted or unknown state. Right next to it lives the silent clean-up step that, when state is fine, quietly resets a story left half-finished by an earlier interrupted run and sends it back through the build, no questions asked.

Why we wrote a guarantee around it, and why that way: the entire reason this story exists is to promise the developer two things at once — the build never starts from a broken baseline, and a normal interrupted-then-resumed run gets picked up automatically without interruption. Those two promises pull against each other, so the contract was written to test both edges: a genuinely bad state must halt and say why, but a legitimately leftover story must be cleaned up and proceed silently. The adversarial review caught it before merge, including catching a residual defect on a re-check that an earlier fix had missed.

Why the outcome is acceptable: the fix throws out the hand-typed list and points the check at the system's single authoritative list of valid story states, dropping the invented entries entirely. The guard now recognizes exactly the states a story can actually be in, so legitimate leftover work resumes silently while genuinely unknown state still halts — the behavior the contract was protecting — and it was re-verified after the fix to confirm the false-alarm path is gone.

▶The build was wired to stop and ask the developer whenever two stories' changes collided during merge — the one thing the system promises it will never do during a run.

What this part of conduct does: as each story finishes building, the orchestrator folds that story's changes back into the shared branch — a merge. Sometimes two stories touch the same lines and the merge collides (a "conflict"). This is the moment the orchestrator decides whether to keep going on its own or to stop and pull the developer in. Why we wrote a guarantee around it, and why that way: the entire reason conduct exists is to collapse a long, nag-heavy build into a single review at the very end — the developer kicks off the run, walks away, and comes back once at the finish line. So the contract pinned down a hard promise: on a normal build the orchestrator asks the developer nothing between start and that final gate, and there is no developer stop anywhere in the merge lifecycle. That promise was written that way because a single mid-build interruption breaks the "let it run and review once" model that justifies the whole rewrite.

Where reality diverged: the keystone scaffold did the opposite. Its per-story merge loop instructed the orchestrator that if a rebase conflict occurred, it should stop and wait for the developer to resolve it. Why the outcome is acceptable: the corrected behavior now matches the promise the contract was protecting — the build never stops to ask during a routine run, and conflicts are handled without the developer — and it was re-verified after the fix, so the no-interruption guarantee is actually enforced in the scaffold the rest of conduct depends on, not merely intended.

▶The files disagreed on who is allowed to flag a finding for the human — a contradiction that, left in, would have let a future change move that authority to the wrong place and quietly break which problems ever reach you.

What a post-merge review is for, in one line: every story passed its own review, but nothing had ever looked at the eight-story stack of merged files all at once to ask the only question that matters here — *do these files still agree with each other?* They did not.

One further integration finding — a structural mismatch in the hand-off shape between the fixer and the Conductor — was not auto-fixed. It was raised to the developer as the report's single Decision, because reconciling it requires an architectural choice rather than a mechanical fix.

04The one decision that needs you

Everything else, the build resolved on its own. This one it deliberately refused to auto-fix, because it's an architecture-level call — exactly the kind DEC-036 says a human should make. (This is the single mid-build alarm firing, working as designed.)

D1 · The fixer and the Conductor don't fully agree on the shape of what they hand each other

In plain terms. The fixer (which acts on problems) and the Conductor (which routes them) pass findings back and forth. The fixer packs its answer one way (details tucked inside a nested envelope); the Conductor was written to read them another way (expecting those details laid out flat on top). They mostly line up — but the specific field that decides "is this dangerous enough to stop the build and ask the human?" sits in the nested envelope, where the Conductor's stop-and-ask check can't see it.

Why it's raised, not fixed. This is the load-bearing seam of the whole safety mechanism, spanning the two main files plus the contract between them — an architecture decision. DEC-036's rule is that the autonomous fixer must not unilaterally pick the shape of a safety-critical contract; a human signs off. (The build already patched the safe half — the everyday paths join up correctly; what's left is to lock in one agreed shape.)

What's actually at stake. conduct fails safe here, not dangerous: if the stop-and-ask field can't be read, a mid-build finding quietly falls through to the end-gate (you still see it) rather than the reverse. So the consequence of leaving it is narrow — the mid-build alarm wouldn't fire from inside the fix loop — not "a dangerous change slips through silently." But it should be made correct before this engine is trusted to run unattended.

Option A (recommended). Teach the Conductor to read the nested envelope and match findings up by their id — extends what the build already started. Smallest change, two files plus the contract. I'd run it as one fix before merge.
Option B. Change the fixer to lay its details out flat, the way the Conductor already reads. Leaves the Conductor untouched but edits the more widely-used builder worker.
Option C. Ship as-is and file a follow-up. The mid-build alarm stays dormant inside the fix loop until it's done; everything else works. Lowest effort now, carries a known gap.

My recommendation: Option A — smallest blast radius, and the build already moved the Conductor halfway there.

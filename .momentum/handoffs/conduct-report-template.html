01Sprint summary

What this sprint set out to do. Replace the legacy intake-queue.jsonl backlog file with an append-only practice ledger (DEC-033), and dissolve the standalone features.json taxonomy into the epic model (DEC-034). The practice ledger is an event log: you never edit or delete a line — you only append events (created, consumed, closed_stale…). To answer "what is still open?" the reader replays every event per item ("folding"): an item that was created then later consumed folds to closed.

What actually shipped. All 9 movements merged independently to sprint/sprint-2026-05-26. The ledger reader, writer, daily stale-close routine, epic-migration tool, and the canvas/greeting surfaces that consume them are all in place and green (785 tests). The AVFL merge-review found no unresolved integration defects. From code review (adversarial bug hunt)

Open/history order was non-deterministic when two events shared a timestamp.

high

a1 · practice_ledger.py → fold_events()

What: events were sorted by timestamp alone. When two events were appended in the same second (common in scripted runs), their relative order — and therefore which one "won" the fold — depended on dict insertion order, which is not guaranteed stable across runs.

Why it matters: the same ledger could fold to a different open set on two consecutive reads — a flaky source of truth. The ledger unit test flaked 3 of 4 runs and was root-caused to exactly this.

sort key was: lambda e: e["ts"] fix: lambda e: (e["ts"], e["event_id"]) # stable tie-break
FIXED & RE-VALIDATED · tie-break on (ts, event_id); ledger test now 5/5 green. No decision needed.

04Decisions needing you

Shared background (both decisions come from the practice ledger, movement a1). a1 replaced the old intake-queue.jsonl with an append-only event log at .momentum/practice-ledger.jsonl: you never edit or delete past lines, you only append events (created, consumed, closed_stale…). To answer "what is still open?" the code reads every event and replays them per item ("folding") — an item that was created then later consumed folds to closed. Both decisions below are about that reader. Neither blocks the sprint — each is "is the current behavior the one we want to bless?"

① Should the ledger reader use DuckDB as the spec says, or keep the pure-Python implementation that was actually written? AC15 / arch D7

The call (the contradiction): the spec — a1's acceptance criterion AC15, plus architecture Decision 52 / DEC-033 D7 — says the reader should use DuckDB, an embeddable SQL engine, to read the JSONL and fold it with SQL. The function is even named _load_duckdb_events. But the code that shipped is plain Python (open file → parse each line → fold in a loop); it imports no DuckDB at all. It is correct, passes all 658 tests, and is arguably better because it has no external dependency (DuckDB would need installing; pure-Python runs anywhere). Nothing is broken — this is purely "which side do we make honest, the spec or the code?"

Recommendation: Bless pure-Python — it is dependency-free, already passing, and DuckDB buys no behavioral gain at this data scale.

The three options & their trade-offs
Bless pure-Python (recommended). Cost: ~5 min — amend AC15 + the two arch decisions to describe the pure-Python fold, and rename _load_duckdb_events → _load_ledger_events. Benefit: zero behavior change, stays dependency-free, the spec stops lying about the implementation.

Implement real DuckDB. Cost: rewrite the reader as DuckDB SQL + add and pin the dependency + CI install step. Benefit: matches the original spec letter-for-letter. Downside: adds a dependency the dev deliberately avoided, for no behavioral gain at this scale.

Defer. Cost: the misleading name and spec linger until a follow-up story is picked up. Benefit: zero work now. Downside: anyone reading _load_duckdb_events meanwhile is misled.

② Should archived (pre-migration) entries be left out of open/history, or backfilled so history spans the cutover? by-design

The call: a2 renamed the old intake-queue.jsonl (old schema: id/status/kind) to practice-ledger-pre-2026-05.jsonl — the archive — and started a fresh ledger with the new event schema (event_id/entity_id/event_type). The reader counts the 88 archived entries (summary shows archive_entries: 88) but does not fold them into open/history, because the old rows are status snapshots, not events, and cannot run through the event-fold logic.

Observable consequence (what a2's QA flagged): history --entity <old-item> shows 1 event (only the new "consumed" a2 appended during cleanup), not 2 — the original "created" lives in the archive, which history does not read. So the chain looks truncated across the migration boundary.

Recommendation: Leave as-is — the archive is a frozen historical snapshot, and a clean migration boundary is worth more than seamless cross-cutover history that nothing currently needs.

The two options & their trade-offs
Leave as-is (recommended). Cost: cross-boundary history is incomplete by design; old entries remain a visible count only. Benefit: clean migration boundary, zero work, new work uses the new ledger cleanly.

Follow-up backfill. Cost: a one-time script reads still-open archive items and emits synthetic new-schema created events. Benefit: open/history show full chains across the old→new cutover. Only worth it if you actually need seamless history across the boundary.

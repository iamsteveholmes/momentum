Background (both decisions come from story a1). a1 replaced the old intake-queue.jsonl with a practice ledger (.momentum/practice-ledger.jsonl) — an append-only event log: you never edit or delete past lines, you only append events (created, consumed, closed_stale…). To answer "what's still open?" the code reads every event and replays them per item ("folding") — an item that was created then later consumed folds to "closed." Both decisions are about that reader. Neither blocks the sprint — each is "is the current behavior the one we want to bless?"

① The practice-ledger reader: DuckDB or pure-Python? AC15 / arch D7

The contradiction: the spec (a1's acceptance criterion AC15, plus architecture Decision 52 / DEC-033 D7) says the reader should use DuckDB — an embeddable SQL engine — to read the JSONL and fold it with SQL. The function is even named _load_duckdb_events. But the code that was actually written is plain Python (open file → parse each line → fold in a loop); it imports no DuckDB at all. It's correct, passes all 658 tests, and is arguably better because it has no external dependency (DuckDB would need installing; pure-Python runs anywhere). Nothing is broken — this is "which side do we make honest?"

The three options & trade-offs
Bless pure-Python (recommended): update AC15 + the two arch decisions to describe the pure-Python fold, and rename _load_duckdb_events → _load_ledger_events. ~5 min of doc + rename, zero behavior change, stays dependency-free.
Implement real DuckDB: rewrite the reader to use DuckDB SQL + add the dependency. Matches the original spec, but adds a dependency the dev deliberately avoided, for no behavioral gain.
Defer: leave the mismatch (misleading name/spec lingers), file a follow-up story.

② Archive entries excluded from open/history derivation by-design

What happened: a1 renamed the old intake-queue.jsonl (old schema: id/status/kind) to practice-ledger-pre-2026-05.jsonl — the "archive" — and started a fresh ledger with the new event schema (event_id/entity_id/event_type). The reader counts the 88 archived entries (summary shows archive_entries: 88) but does not fold them into open/history — only new-schema events get replayed, because the old rows are status snapshots, not events, and can't run through the event-fold logic.

Visible consequence (what a2's QA flagged): history --entity <old-item> shows 1 event (just the new "consumed" a2 appended during cleanup), not 2 — the original "created" lives in the archive, which history doesn't read. So the chain looks truncated across the migration boundary.

The two options & trade-offs
Leave as-is (recommended): the archive is a frozen historical snapshot; new work uses the new ledger; old entries remain a visible count. Clean migration boundary; cross-boundary history is incomplete by design.
Follow-up backfill: a one-time script reads still-open archive items and emits synthetic new-schema created events so they appear in open/history with full chains. More work; only worth it if you need seamless history across the old→new cutover.
